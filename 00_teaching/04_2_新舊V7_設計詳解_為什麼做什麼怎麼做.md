# 新舊 V7 設計詳解：為什麼、做什麼、怎麼做

> 目的：把「舊 V7（歷史版本：JsonFile + userId:number）」與「新 V7（目前實作：userId:string + Auth/Store 分離）」的差異講清楚，讓升級到 V8/V9 時不會迷路。

---

## 1. 問題背景（為什麼要重建新 V7）

舊 V7 可以運作，但在「要繼續升級到 V8/V9」時，會出現三個長期成本：

1. 身分責任與資料責任綁在同一層

- `Store` 同時做 `login/getUserById` 與訂單資料存取。
- 一旦登入方案更動，`Store` 也要跟著改。

2. `userId:number` 對接 Better Auth 成本高

- Better Auth 常見是字串主鍵（例如 UUID 或 CUID）。
- 若 V7/V8 還維持 number，V9 會發生跨層型別翻轉。

3. 路由層耦合過多細節

- route 同時承擔認證與資料層決策，難以替換底層實作。

一句話：舊 V7 能跑，但不是「可長期演進」的地基。

---

## 2. 架構對照（做了什麼）

### 舊 V7（可運作版）

1. 核心特徵

- 儲存：JsonFile
- 身分：由 Store 直接處理（`login/getUserById`）
- ID：`userId:number`

2. 優點

- 上手快、容易 demo。
- 不需要外部服務即可跑全流程。

3. 限制

- Auth 與業務資料耦合。
- 升級到 PostgreSQL / Better Auth 時改動面積大。

### 新 V7（重建地基版）

1. 核心特徵

- 儲存：仍可先用 JsonFile（降低遷移阻力）
- 身分：抽出 `Auth` 模組
- ID：從起點就固定 `userId:string`

2. 具體改動

- 合約層：`User.id`、`Order.userId` 改為 string。
- 介面層：`Store` 移除 `login/getUserById`。
- 模組層：新增 `auth/Auth.ts`、`auth/DemoAuth.ts`、`auth/index.ts`。
- route 層：登入找 `auth`，訂單找 `store`。
- 資料層：`data/store.json` 已完成字串化（`users[].id`、`orders[].userId` 皆為 string）。

3. 效果

- V7 -> V8：可只替換 store 實作（Json -> Postgres）。
- V8 -> V9：可只替換 auth 實作（DemoAuth -> Better Auth）。

---

## 3. 重建順序（怎麼做）

建議固定使用以下順序，避免半套狀態：

1. 先改 contract truth

- 先把 `userId` 定義為 string，讓上游事實一致。

2. 再改 Store 介面

- 移除 auth 責任，只保留 menu/orders。

3. 補 Auth 抽象與實作

- 建立 `Auth` 介面與 `DemoAuth`。

4. 調整 Store 實作

- Json store 專注業務資料。
- 載入歷史舊資料時可把 number userId normalize 成 string（相容性策略）；目前主資料檔已是 string。

5. 最後改 backend route

- route 只做協調，不混入底層細節。

---

## 4. 舊 V7 到新 V7 的對應思考

1. 舊 V7 不是錯，而是階段任務不同

- 它完成了「先跑起來」的教學目標。

2. 新 V7 是演進型版本

- 目標是「未來可替換、可擴充、可升級」。

3. 兩者應該並存於教材

- 舊 V7：示範最小可行與快速回饋。
- 新 V7：示範邊界治理與長期演進。

## 5. 決策原則與工程治理

這一節把新 V7 重建過程中涉及的幾個核心決策原則系統化整理，它們不只適用於 V7，也是後續每一輪重構的底層思維框架。

### 原則一：型別語意精確，名字即文件

以 login 回應的型別為例：

**舊做法（結構性描述）**

```ts
type SafeUser = Omit<User, "password">;
```

這個名字描述的是「User 去掉某個欄位」，讀者要自己推算它代表什麼。

**新做法（語意描述）**

```ts
// shared/contracts.ts
export type SessionUser = { id: string; email: string; name: string };
```

這個名字直接說明「可以安全暴露給前端的使用者身份」，不需要任何額外說明。

**效果**：型別名稱本身就是文件。閱讀程式碼時，不需要心算「這兩個型別到底是不是同一件事」。

---

### 原則二：single source of truth 讓整條資料流只有一個型別

以 login 這條資料流為例，重構後從頭到尾只有一個型別：

| 位置                       | 型別                           |
| -------------------------- | ------------------------------ |
| 後端 `Auth.login()` 回傳   | `SessionUser`                  |
| 後端 login route 回傳      | `ApiDataResponse<SessionUser>` |
| 前端 `handleLogin` 接收    | `ApiDataResponse<SessionUser>` |
| 前端 `user` state          | `SessionUser \| null`          |
| 前端 localStorage 還原驗證 | `SessionUser` 形狀             |

`shared/contracts.ts` 作為 single source of truth 的核心效益是：**型別只定義一次，前後端各自 import，重構時只需改一個地方。**

---

### 原則三：需求改變時，三層同步更新

常見的錯誤做法是：只改 contracts.ts 和 API 邏輯，但不改資料本身。

例如 `userId` 從 `number` 改為 `string` 時，若 `data/store.json` 裡的舊格式沒有一起更新，就必須在讀取層加入相容性函式：

```ts
function normalizeUserId(raw: unknown): string {
  if (typeof raw === "number") return String(raw);
  if (typeof raw === "string") return raw;
  return "";
}
```

這段程式碼的存在，代表系統裡有兩個「真相」同時活著，讀者需要同時理解「現在的格式」和「過去的格式」。

**正確做法：需求改變時，三層同時更新**

| 層次                  | 範例                               |
| --------------------- | ---------------------------------- |
| contracts.ts 型別定義 | `id: string`                       |
| API / 程式邏輯        | 不再 `Number(id)` 轉型             |
| 資料本身              | `data/store.json` 的 id 全改為字串 |

一次性修正資料的成本，遠低於長期維護相容層的累積成本。本專案正是這樣做的：`data/store.json` 的 userId 已全面改為字串，因此 `normalizeUserId` 與 `parseStoredSessionUser` 可以安全移除。

> **何時保留相容層是合理的**：外部資料來源無法控制（例如第三方 API）、資料量太大遷移成本確實高、或多系統共用同一份資料時。若資料在自己掌控範圍內，幾乎都可以透過一次性修正來消除相容層。

---

### 原則四：防禦性程式碼量剛好

移除不再必要的相容層後，剩下的邊界驗證應該只對應**真正存在的風險**：

```ts
// 這是真正的邊界防禦：localStorage 資料可能損毀
const parsed = JSON.parse(savedUser) as SessionUser;
if (
  typeof parsed?.id === "string" &&
  parsed.id.trim() !== "" &&
  typeof parsed?.email === "string" &&
  typeof parsed?.name === "string"
) {
  setUser(parsed);
} else {
  window.localStorage.removeItem(USER_STORAGE_KEY);
}
```

這段驗證的每一行都有對應的真實風險（JSON 可能損毀、欄位可能缺失）。它不是為了「防萬一」而存在，而是為了「已知的邊界情境」而存在。

**判斷標準**：如果你說不出「這段程式碼在防禦什麼具體情境」，它可能就是技術債。

---

### 原則五：`main` 永遠是可信任的穩定狀態

本專案採用一個心智負擔最小的分支策略：

> `main` = 完全測試通過、可直接使用的版本

| 分支     | 語意                 |
| -------- | -------------------- |
| `main`   | 通過驗證的里程碑版本 |
| `feat/*` | 進行中，不保證穩定   |

工作流程：在 feature branch 上開發 → 確認功能完整、型別正確、build 成功 → merge 回 `main`。

這個策略與前三個原則一脈相承：讓系統的狀態真實反映當前事實，而不是讓歷史狀態繼續混淆現在。對這個教學專案來說，`main` 指向「目前教到哪、測到哪」，學生 clone 後不需要猜該切哪個分支。

## 7. 後端設計品質決定前端能做什麼

2025–2026 年的前端框架演進（Remix 3、TanStack Start、SvelteKit）有一個共同方向：**把「讀」與「寫」變成框架等級的一階概念**。

具體來說：

- 讀取（loader / data function）→ 對應後端 `GET` route
- 寫入（action / mutation）→ 對應後端 `POST / PATCH / DELETE` route
- Single Flight Mutations → 框架自動 dedup、合併 refetch，不再需要手刻 debounce

這些能力能不能真正發揮，取決於後端 API 的設計品質：

| 後端做到                                      | 前端才能做到                                                           |
| --------------------------------------------- | ---------------------------------------------------------------------- |
| 型別在 contracts.ts 有 single source of truth | Server Functions 可以自動傳遞型別安全，不需要手寫 fetch + 型別斷言     |
| GET / POST / PATCH 語意清晰                   | 框架能推斷哪些請求可快取、哪些需要 invalidate                          |
| 資料格式穩定（不需相容層）                    | 前端可以用最薄的方式接收資料，不需要防禦性轉型                         |
| 邊界防禦集中在後端                            | 前端只需驗證真正的邊界（如 localStorage 還原），不需要到處寫 normalize |

換句話說：新一代前端框架的目標是「盡量變薄，把 Web 標準穩固暴露出來」。但如果後端 API 是不穩定的、型別是分散的、資料格式是雙重真相的，框架的薄層設計就沒有地方站穩。

> **這門課正在做的事，就是在替未來的同構框架、Server Functions、甚至 AI 生成前端打好地基。**  
> `contracts.ts` + 清晰的 route 語意 + 資料三層同步，是後端對前端最重要的貢獻。

1. 本倉庫目前新 V7 的主資料檔 `data/store.json`，`users[].id` 與 `orders[].userId` 已是 string。
2. 教材中提到的 `userId:number` 是「歷史舊 V7 脈絡」，用來說明為何需要這次重建。
