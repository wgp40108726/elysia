# V7 → V8 → V9 重建升級路徑：實作清單與紀錄（新主線）

> 分支：feat/v7-clean-auth-store
> 原則：先打穩新 V7 的分層與型別地基，再往新 V8、新 V9 升級。

---

## 已確認的設計決策

1. `userId` 從新 V7 起固定為 `string`。
2. `userId` 視為 opaque string；`"0001"` 只是示例格式。
3. `Auth` 與 `Store` 分離：
   - `Auth`：登入與使用者身分查找
   - `Store`：menu/orders/order_items 業務資料存取
4. `Route` 只協調 `Auth` 與 `Store`，不混入底層細節。

---

## 這一步的背後邏輯（新 V7 為什麼要先做這件事）

新 V7 不是單純把舊程式碼改一點點，而是在重建一條之後可以一路升到新 V9 的乾淨主線。

這一步先做 `Auth` / `Store` 分離，有三個原因：

1. 避免責任混層

- 舊版本把 `login()`、`getUserById()` 這類身份責任放在 `Store`
- 這會讓「資料存取」與「認證機制」糾纏在一起
- 一旦未來更換登入方式，Store 就會被迫一起改

2. 把可變的東西隔離起來

- 未來 V7 → V8 會從 JsonFile 進到 DB
- V8 → V9 會從 DemoAuth 進到 Better Auth
- 若 `Auth` 與 `Store` 先分離，後面每次升級只需要替換其中一層，不用整條重寫

3. 把 `userId` 型別穩定下來

- 如果新 V7 還沿用 number，到了新 V9 又會重演一次型別翻轉
- 因此新主線從一開始就固定 `userId: string`
- 之後就算從 `"0001"` 換成 UUID，也只是值格式改變，不是型別改變

一句話總結：

> 新 V7 先解決的是「架構邊界」與「型別穩定性」，不是先追求更多功能。

---

## 這一步的實作順序（為什麼照這個順序做）

這一輪的順序不能亂，因為每一步都依賴前一步的結果。

1. 先改契約與型別

- `shared/contracts.ts`
- 先把 `User.id`、`Order.userId` 固定成 `string`
- 這是最上游的事實

2. 再改 `Store` 介面

- `store/Store.ts`
- 先把 `login()`、`getUserById()` 從介面移除
- 再把所有 `userId` 參數改成 `string`

3. 接著補 `Auth` 抽象

- `auth/Auth.ts`、`auth/DemoAuth.ts`、`auth/index.ts`
- 讓原本屬於 Store 的登入責任有地方可去

4. 再改 `Store` 的具體實作

- `store/json/JsonFileStore.ts`
- 把 auth 方法刪掉，保留業務資料存取
- 同時把舊資料中的數字 userId normalize 成字串

5. 最後才改 route

- `backend.ts`
- route 改為協調 `auth` 與 `store`
- 也就是：登入找 `auth`、訂單資料找 `store`

如果順序反過來，例如先改 route、但 `Auth` 還不存在，或 `Store` 介面還沒清乾淨，整體就會卡在半套狀態。

所以這一步的正確順序是：

```text
contracts → Store interface → Auth abstraction → Store implementation → route
```

## 實作紀錄 #002（運行環境收斂與故障排查）

- 日期：2026-04-23
- 背景：同時存在 `00_demo01` 與舊 worktree（如 `00_demo01_v7_clean`）執行紀錄，導致排查訊號混雜。

### 觀察到的症狀

1. `ENOENT: ./public/index.html`、`ENOENT: .../public`
2. 前端偶發「加入購物車失敗」，但 API 有時又看起來正常
3. 終端出現 `exit code 137/143`、`Terminated`

### 根因判斷

1. 不是單一程式邏輯錯誤，而是「執行環境混線」：

- 多個 backend 行程同時存在
- 不同 cwd（含舊 worktree）都可能監聽 3000

2. `public/` 缺失屬於執行模式/目錄問題：

- 沒先 build frontend
- 或在非目標專案目錄啟動

### 收斂解法（標準流程）

1. 只保留單一來源目錄：固定使用 `00_demo01`
2. 清掉舊行程，避免 3000 被不同版本 backend 搶占：

```bash
fuser -k 3000/tcp || true
pkill -f "bun backend.ts" || true
pkill -f "bun run dev" || true
```

3. 重新產生整合前端資產：

```bash
bun run build:frontend
```

4. 只啟動一個 backend：

```bash
bun --watch backend.ts
```

5. 以 smoke test 驗證而非只看瀏覽器表象：

- `GET /`
- `GET /health`
- `POST /api/auth/login`
- `GET /api/orders/current?userId=0001`
- `PATCH /api/orders/:id`

### 教學結論（要寫進課堂口訣）

1. `dev` 模式看 `5173`，整合模式看 `3000`（前提是先 build frontend）。
2. 看到 `ENOENT public/index.html` 先查「目錄與 build 狀態」，不要先懷疑業務邏輯。
3. 看到 API 時好時壞，先查「是否有多 backend 行程」再查程式碼。

## 實作紀錄 #005（feat/v8-namespace 實際執行的踩坑與驗證）

- 日期：2026-04-25
- 分支：`feat/v8-namespace`
- commit：`86cc9b3`

### 執行流程全記錄

**Step 1：修改 `db/schema.ts`**

```ts
// 改前
import { pgTable, ... } from "drizzle-orm/pg-core";
export const usersTable = pgTable("users", { ... });

// 改後
import { pgSchema, ... } from "drizzle-orm/pg-core";
const appSchema = pgSchema("breakfast");
export const usersTable = appSchema.table("users", { ... });
```

只改四行 import 與四個 `pgTable` → `appSchema.table`，其餘完全不動。

**Step 2：`bun run db:generate`**

Drizzle Kit 偵測到資料表從 `public` 移入 `breakfast`，詢問是「create 新表」還是「rename 舊表」。選擇「create table」（在新 namespace 全新建立），產生：

```sql
CREATE TABLE "breakfast"."users" ( ... );
CREATE TABLE "breakfast"."menu_items" ( ... );
CREATE TABLE "breakfast"."orders" ( ... );
CREATE TABLE "breakfast"."order_items" ( ... );
DROP TABLE "menu_items" CASCADE;
DROP TABLE "order_items" CASCADE;
DROP TABLE "orders" CASCADE;
DROP TABLE "users" CASCADE;
-- 加上 FK 與 unique index
```

**Step 3：執行 migration — 踩坑**

```bash
bun run db:migrate  # → 失敗，exit code 1，但無詳細錯誤訊息
```

這是 `drizzle-kit migrate` 搭配 `@neondatabase/serverless` 在某些版本的已知問題：命令靜默失敗。

**解法**：直接手動執行 SQL：

```ts
// 拆成個別 statement，逐條執行並印出結果
const statements = sql.split("--> statement-breakpoint");
for (const stmt of statements) {
  await pool.query(stmt); // 11 條全部 OK
}
```

> **教學結論**：Migration 工具不是黑盒子。當 `db:migrate` 失敗時，直接讀 SQL 檔案手動執行，這個能力比盲目重試更重要。

**Step 4：`bun run db:migrate-json --reset` — 再次踩坑**

Seed 腳本 `scripts/migrate-json-to-db.ts` 裡有兩處**硬寫的 SQL 字串**沒有 schema prefix：

```ts
// 改前（會找不到資料表）
sql`truncate table order_items, orders, menu_items, users ...`;
sql`select setval('users_id_seq', ...)`;

// 改後（加上 breakfast. prefix）
sql`truncate table breakfast.order_items, breakfast.orders, ...`;
sql`select setval('breakfast.users_id_seq', ...)`;
```

**為什麼 INSERT 不需要改？** 因為 INSERT 用的是 Drizzle table 物件（`usersTable`），Drizzle 已知道它屬於 `breakfast` schema，自動加 prefix。只有**裸字串 SQL** 才需要手動加。

結果：2 users / 12 menu / 5 orders / 11 order_items 全部成功入庫。

**Step 5：啟動後端驗證**

```bash
curl http://localhost:3000/health
# → {"status":"ok"}

curl -X POST http://localhost:3000/api/auth/login \
  -d '{"email":"demo@example.com","password":"1234"}'
# → {"data":{"id":1,"email":"demo@example.com","name":"示範使用者"}}
```

**`PgStore.ts`、`db/client.ts`、`backend.ts` 全部零改動。**

---

### 結論：三個教學重點

1. **分層設計的威力**：改了資料庫的 namespace，API 層完全不知道。這不是巧合，而是「`PgStore` 只依賴 Drizzle table 物件，不依賴資料表名稱字串」的設計決策帶來的保護。

2. **硬寫 SQL 是邊界破口**：`scripts/migrate-json-to-db.ts` 的 `truncate` 與 `setval` 需要手動更新，正是因為它們繞過了 Drizzle ORM 的抽象層，直接操作底層字串。這是「raw SQL 的代價」，也是選擇用 ORM 的原因之一。

3. **Migration 工具可能無聲失敗**：了解 migration SQL 的內容，並知道如何手動執行，是後端工程師的基本能力，不能只依賴工具指令成功。

## 實作紀錄 #006（Expand-Contract 模式：正確的 migration 策略）

- 日期：2026-04-25
- 分支：`feat/v8-namespace`

### 問題：我們刪了不該刪的東西

`drizzle-kit generate` 在選擇「create table」（而非 rename）時，自動產生了 DROP TABLE：

```sql
DROP TABLE "menu_items" CASCADE;
DROP TABLE "order_items" CASCADE;
DROP TABLE "orders" CASCADE;
DROP TABLE "users" CASCADE;
```

這些語句被手動執行後，**public schema 下的四張舊資料表消失了**。這導致：

- 舊 V8（`legacy/v8/`）連同一個 DB 時找不到資料表
- 無法比較新舊行為
- 一旦新版有 bug，沒有舊版資料可以回退

### 正確策略：Expand-Contract 模式

這在業界有個標準名詞，分兩個階段：

**Phase 1 — Expand（擴張）**
只建立新的 schema，**完全不刪舊的**。新舊並存：

```sql
CREATE SCHEMA "breakfast";
CREATE TABLE "breakfast"."users" ( ... );
CREATE TABLE "breakfast"."menu_items" ( ... );
-- 加上 FK / index
-- ❌ 不要 DROP TABLE "users" CASCADE;
```

**Phase 2 — Contract（收縮）**
等新版穩定跑了一段時間，確認：

- 所有 API 驗證通過
- 資料遷移完整
- 不再需要回退

才建一個**獨立的 migration** 執行刪除：

```sql
-- 0002_cleanup_public_tables.sql（獨立執行，確認穩定後才跑）
DROP TABLE IF EXISTS "public"."order_items" CASCADE;
DROP TABLE IF EXISTS "public"."orders" CASCADE;
DROP TABLE IF EXISTS "public"."menu_items" CASCADE;
DROP TABLE IF EXISTS "public"."users" CASCADE;
```

### 本次補救

發現問題後，手動在 Neon 重建了 public schema 的四張資料表（使用 `0000_nervous_ronan.sql` 的內容），並從 `0001_orange_micromacro.sql` 移除四條 DROP TABLE，改為加上說明註解。

這使得：

- `breakfast` schema：新版 V8 namespace 使用，目前有資料
- `public` schema：舊版 V8 結構還原，供比較與備援

### 選擇代價對照

| 策略                      | 優點               | 代價                   |
| ------------------------- | ------------------ | ---------------------- |
| 直接 DROP（方向 A）       | DB 乾淨            | 無法回退，新舊無法比較 |
| Expand-Contract（方向 B） | 可回退，可並行比較 | 多佔一點 DB 空間       |

> **原則**：在正式環境，幾乎永遠選方向 B。DB 空間便宜，資料和回退能力無價。

### 教學口訣

> DROP TABLE 放到獨立 migration，等確認穩定再砍。
> Expand 先，Contract 後，這順序不能倒。

## 架構決策紀錄（ADR）：版本升級時的資料庫策略

**決策**：進版時，舊資料庫不應更動。新版另建 namespace，在其中建立新的資料表。

**理由**：

1. **舊版仍可運作**：舊 schema（`public`）完全不動，舊版程式碼隨時可切回並正常執行，不需要 rollback 程式碼就能驗證問題出在哪一層。

2. **新舊可並行比較**：開發期間可以同時跑舊版與新版，直接比較 API 回應，確認行為一致。

3. **零風險的升級路徑**：問題發生時，只需把連線指向舊 schema，不需要做任何資料還原。

**實作方式**：

```
舊版（V8）          新版（V8 + namespace）
─────────────────   ─────────────────────────
public.users        breakfast.users
public.menu_items   breakfast.menu_items
public.orders       breakfast.orders
public.order_items  breakfast.order_items
```

- 程式碼層：只改 `db/schema.ts` 的 `pgTable` → `pgSchema("breakfast").table`
- 舊版程式碼（`legacy/v8/`）連到 `public` schema，完全不動
- 新版 migration 只做 `CREATE TABLE`，不做 `DROP TABLE`
- 確認穩定後，才另建 `0002_cleanup_public_tables.sql` 執行刪除

**何時才能刪舊表**：

- 新版 API 已在正式環境穩定運行一段時間
- 所有資料已驗證正確遷移
- 團隊確認不需要回退

> **核心原則**：進版的代價是 DB 空間，回退失敗的代價是服務中斷與資料損失。前者幾乎永遠值得。

## 實作紀錄 #007（舊 V8 vs 舊 V8 namespace 版：差異與資料路徑判讀）

- 日期：2026-04-25
- 對照分支：
  - 舊 V8（無 namespace）：`feat/v8-drizzle-neon`
  - 舊 V8 namespace 版（目前）：`feat/v8-namespace`

### 一、兩個版本的核心差異

| 面向               | 舊 V8（無 namespace）             | 舊 V8 namespace 版（目前 branch）      |
| ------------------ | --------------------------------- | -------------------------------------- |
| Drizzle table 定義 | `pgTable("users")`（預設 public） | `pgSchema("breakfast").table("users")` |
| SQL 實際表名       | `public.users` / `public.orders`  | `breakfast.users` / `breakfast.orders` |
| API 路由程式       | 不變                              | 不變                                   |
| Store 介面         | 不變                              | 不變                                   |
| 風險控制           | 容易與主線資料混用                | 可與舊表並存，比較與回退較安全         |

重點：這次改造主要是「資料庫命名空間切換」，不是「API 行為重寫」。

### 二、程式中決定「用 DB 還是用 JSON」的路徑

資料來源不是看 namespace，而是看 `STORE_DRIVER`。判讀順序如下：

1. `backend.ts` 建立 store

```ts
const store = createStore({ dataFilePath: "./data/store.json" });
```

2. `store/index.ts` 用 `process.env.STORE_DRIVER` 選 driver

```ts
export function createStore(options: CreateStoreOptions = {}): Store {
  const driver = process.env.STORE_DRIVER;

  if (driver === "json") {
    return new JsonFileStore({
      dataFilePath: options.dataFilePath ?? "./data/store.json",
    });
  }

  if (driver === "postgres") {
    return new PgStore({
      dataFilePath: options.dataFilePath ?? "./data/store.json",
    });
  }

  return new JsonFileStore({
    dataFilePath: options.dataFilePath ?? "./data/store.json",
  });
}
```

3. 若是 `postgres`，才會進一步由 `db/schema.ts` 與 `PG_SCHEMA` 環境變數決定 schema

```ts
// db/schema.ts（目前 branch 的寫法）
const appSchema = pgSchema(process.env.PG_SCHEMA ?? "public");
export const usersTable = appSchema.table("users", { ... });
```

### 三、關鍵觀念（避免常見誤解）

1. `STORE_DRIVER` 決定「資料引擎」：PostgreSQL 或 JSON file。
2. `PG_SCHEMA` 決定「PostgreSQL 內的 namespace」，只在 `STORE_DRIVER=postgres` 時生效。
3. 所以：
   - `STORE_DRIVER=json` 時，`PG_SCHEMA` 完全無作用，程式仍走 JSON。
   - `STORE_DRIVER=postgres` + `PG_SCHEMA=breakfast` → 走 `breakfast.*`（namespace 版）。
   - `STORE_DRIVER=postgres` + `PG_SCHEMA=public` 或不設定 → 走 `public.*`（舊 V8 行為）。

### 四、目前 branch 的實際判定

- `.env` 設定：`STORE_DRIVER=postgres` + `PG_SCHEMA=breakfast`
- `db/schema.ts`：讀取 `PG_SCHEMA`，動態建立 `pgSchema("breakfast")`

=> 結論：目前 branch 會使用 PostgreSQL 的 `breakfast` namespace，不是 JSON file。

## 實作紀錄 #008（PG_SCHEMA 環境變數化 + Render.com 部署注意事項）

- 日期：2026-04-25
- 分支：`feat/v8-namespace`
- commit：`6a862b6`

### 為什麼要環境變數化

namespace 硬寫在程式裡（`pgSchema("breakfast")`）時：

- 切換 schema 需要改程式碼
- 不同環境（dev / staging / prod）無法各自指向不同 namespace
- 程式碼與環境配置混在一起，違反十二因素應用（12-factor app）原則

改為 `pgSchema(process.env.PG_SCHEMA ?? "public")` 後：

- 切換只需改環境變數，程式碼不動
- `seed` 腳本的 truncate / setval 也全部動態化，不再硬寫 schema 名稱

### 修改點清單

**`db/schema.ts`**（1 行）

```ts
// 改前
const appSchema = pgSchema("breakfast");

// 改後
const appSchema = pgSchema(process.env.PG_SCHEMA ?? "public");
```

**`scripts/migrate-json-to-db.ts`**（truncate + setval x4，共 5 處）

```ts
// 改前（硬寫）
sql`truncate table breakfast.order_items, breakfast.orders, ...`;
sql`select setval('breakfast.users_id_seq', ...)`;

// 改後（動態）
const schema = process.env.PG_SCHEMA ?? "public";
sql.raw(`truncate table ${schema}.order_items, ${schema}.orders, ...`);
sql.raw(`select setval('${schema}.users_id_seq', ...)`);
```

**`.env.example`**（新增說明）

```
STORE_DRIVER=postgres
# PostgreSQL namespace（schema）：切換 public（舊 V8）或 breakfast（namespace 版）
PG_SCHEMA=breakfast
```

### 切換對照表

| 目標            | STORE_DRIVER | PG_SCHEMA     | 說明                  |
| --------------- | ------------ | ------------- | --------------------- |
| 舊 V8（public） | postgres     | public 或不設 | 連到 public schema    |
| Namespace 版    | postgres     | breakfast     | 連到 breakfast schema |
| 離線開發        | json         | 不需要        | 走 JSON file，不連 DB |

### Render.com 部署注意事項

部署到 Render.com 時，**所有 `.env` 的環境變數都必須在 Render Dashboard 手動設定**，因為 `.env` 不會進入 git（也不應該進入 git，以免洩漏金鑰）。

**必填的環境變數清單（以 namespace 版為例）：**

| 變數名稱                 | 範例值             | 說明                                         |
| ------------------------ | ------------------ | -------------------------------------------- |
| `DATABASE_URL`           | `postgresql://...` | Neon pooler 連線字串（runtime 用）           |
| `DATABASE_URL_MIGRATION` | `postgresql://...` | Neon 直連字串（migration 用）                |
| `STORE_DRIVER`           | `postgres`         | 資料引擎，必須設為 postgres                  |
| `PG_SCHEMA`              | `breakfast`        | ⚠️ 新增！忘記設定就會走 `public`（fallback） |
| `PORT`                   | `3000`             | Render 通常會自動注入，可不設                |

> ⚠️ **最容易遺漏的就是 `PG_SCHEMA`**。因為它有 fallback（`?? "public"`），不設定不會報錯，但程式會連到 `public` schema，讓人以為部署成功，實際上走的是舊表。這是「靜默錯誤」的典型案例。

**建議的驗證步驟（部署後）：**

```bash
# 1. 確認 API 正常
curl https://your-service.onrender.com/health

# 2. 確認登入走的是 breakfast.users（非 public.users）
curl -X POST https://your-service.onrender.com/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"demo@example.com","password":"1234"}'
# 若回傳 user 資料 → 確認 PG_SCHEMA 連線正確

# 3. 若懷疑走錯 schema，可在 Neon console 執行
SELECT schemaname, tablename FROM pg_tables
WHERE tablename = 'users'
ORDER BY schemaname;
-- 應看到 breakfast.users 有資料，public.users 可為空
```

### 教學口訣

> `.env` 是本地秘密，Render Dashboard 是雲端秘密。兩者內容要同步，但永遠不要把 `.env` 推上 git。
> 有 fallback 的環境變數特別危險：不設定不報錯，但行為偷偷改變。

## 實作紀錄 #009（feat/v8-clean-drizzle-neon：新主線 V8 導入 Drizzle + Neon）

- 日期：2026-04-26
- 分支：`feat/v8-clean-drizzle-neon`（從 `feat/v7-clean-auth-store` 切出）

### 目標

在新 V7 已打穩的 Auth/Store 分層地基上，將 `Store` 與 `Auth` 的底層從 JSON file 升級為 Drizzle + Neon PostgreSQL，**同時維持 `backend.ts`、`shared/contracts.ts`、`Auth` 介面、`Store` 介面全部零改動**。

### 背後的決策邏輯

#### 1. 為什麼從 V7 切出而不從 V8-namespace 切出？

`feat/v8-namespace` 是「舊 V8 加 namespace 改造」的教學示範分支，架構上 `userId` 還是 `number`、`Auth` 與 `Store` 未分離。它的目的是讓學生看到「只改 schema.ts 就能加 namespace」，不是新主線的起點。

`feat/v7-clean-auth-store` 才是新主線地基：

- `userId` 全面 `string`
- `Auth` / `Store` 責任完全分離
- `shared/contracts.ts` 是前後端唯一事實來源

所以新 V8 從 V7 切出，而非從舊 V8 切出。

#### 2. 為什麼 Auth 也要換成 PgAuth？

V8 換成 PostgreSQL 後，`orders` 有 `user_id` 外鍵指向 `users` 表。若 `Auth` 繼續查 JSON file，兩個資料來源就脫鉤：

- JSON 有的 user，PG 裡不一定存在
- `user_id` FK 驗證會失敗

因此 `STORE_DRIVER=postgres` 時，`Auth` 也必須同步切換到 `PgAuth`（查 `users` table）。

#### 3. 為什麼 `SessionUser` 是唯一對外身份型別？

`shared/contracts.ts` 是前後端的 single source of truth。`SessionUser` 定義了 `{ id: string, email: string, name: string }`，這是 Auth 認證後唯一對外暴露的使用者型態，`password` 永遠不出邊界。`PgAuth` 的 `login()` 和 `getUserById()` 回傳型別永遠是 `SessionUser`，不是 DB row。

#### 4. 為什麼 Drizzle schema 要對照 contracts.ts 設計？

DB 欄位命名以 `shared/contracts.ts` 為準（snake_case 版），不另外發明。讓欄位名稱和合約型別一眼對應，降低心智負擔：

| contracts.ts         | DB 欄位      |
| -------------------- | ------------ |
| `MenuItem.image_url` | `image_url`  |
| `Order.userId`       | `user_id`    |
| `Order.createdAt`    | `created_at` |

#### 5. 為什麼 namespace 選 `bf_v8`？

- `public`：PG 預設，禁用
- `breakfast`：`feat/v8-namespace` 教學示範用，已有資料
- `v8_legacy`：`legacy/v8` 舊版用
- `drizzle`：drizzle-kit 內部用

`bf_v8` = breakfast app + v8，清楚識別版本與專案。未來 V9 升級時，V8 的 `bf_v8` schema 保留供對照（Expand-Contract 原則），V9 另建 `bf_v9`。

#### 6. 每個分支有自己的 DB schema 原則

> 每個 feature branch 對應一個獨立的 PG namespace，互不干擾，隨時可切換比較，也可獨立部署驗證。

### 優點

1. **分層保護徹底**：換底層（JSON → PG）不影響任何上層介面，API 合約不變
2. **Auth/Store 資料來源一致**：`STORE_DRIVER` 同時切換，不會有 JSON users / PG orders 脫鉤的問題
3. **每個分支獨立 DB namespace**：可以同時跑不同版本對照，互不干擾
4. **contracts.ts 對照 DB schema**：命名一致，降低心智負擔
5. **seed 自動化**：`PgStore.init()` 內建 `seedFromJsonIfEmpty()`，migration 後第一次啟動自動匯入資料

1. **userId 遵循 contracts.ts 事實**：DB schema 的 `users.id` 和 `orders.user_id` 都用 `text` 型別，完全對應 `contracts.ts` 的 `string` 定義。不做任何隱性型別轉換，簡潔且無歧義，同時支援任意 userId 格式（numeric、UUID、email 等）。

3. **`seedFromJsonIfEmpty()` 只在 DB 為空時執行**：若 migration 後 DB 已有資料（例如重複執行），seed 不會覆蓋，需手動 truncate 再重啟才能重新 seed。

## 實作紀錄 #010（feat/v8-clean-drizzle-neon-v2：Schema 一致性設計 - Zod 統一 API 合約層）

- 日期：2026-04-26

### 目標

在 #009 已完成的 Drizzle + Neon 底層地基上，解決 **schema 定義散落三處、需手動同步** 的問題：

- `shared/contracts.ts`（TypeScript interface，純型別）
- `backend.ts`（Elysia `t.Object()`，手動鏡像 contracts.ts）
- `db/schema.ts`（Drizzle table，DB 結構）

目標：讓 `shared/contracts.ts` 成為 **API 業務層的唯一真相來源**，`backend.ts` 只做組合，不再重複定義。

**環境不變**：`.env` 不動、`db/schema.ts` namespace `bf_v8` 不動、Auth/Store 實作不動。

### 背後的決策邏輯

#### 1. 為什麼不用 Drizzle schema 作為 Single Source？

技術上 `$inferSelect` 可以推導型別，但 DB schema 包含 `password`、`created_at` 等 DB 內部欄位，而 API contract 應該只暴露業務欄位（`SessionUser` 不含 `password`）。

混用會造成：

- 安全風險：不小心把 `password` 暴露在 API response
- 演進困難：DB 加欄位時，強制 API 跟著改
- 責任不清：「什麼是 DB 結構考量」vs「什麼是 API 業務考量」模糊

**正確的分層**：

| 層           | 工具                         | 用途                       | 是否含 password |
| ------------ | ---------------------------- | -------------------------- | --------------- |
| API 業務合約 | Zod（`shared/contracts.ts`） | 前後端共用型別、運行時驗證 | ❌              |
| API 路由     | Elysia + Zod（`backend.ts`） | Route 驗證、OpenAPI 生成   | ❌              |
| DB 儲存      | Drizzle（`db/schema.ts`）    | 資料庫結構與型別推導       | ✅              |

#### 2. 為什麼選 Zod 而不是繼續用 Elysia t？

| 考量                         | Elysia `t`（TypeBox）                      | Zod                                              |
| ---------------------------- | ------------------------------------------ | ------------------------------------------------ |
| 能放在 shared/contracts.ts？ | 可以，但 Elysia 是後端套件，不應進入共用層 | ✅ 可以，純語言層工具                            |
| frontend 也能 import 用？    | ❌ 前端沒有 Elysia                         | ✅ frontend 可以直接 import 驗證                 |
| 型別推導                     | 需要手動 type                              | `z.infer<...>` 自動推導                          |
| Elysia 支援？                | 原生                                       | ✅ Standard Schema V1 介面，Elysia 1.4+ 直接支援 |

#### 3. 為什麼兩個檔案的改動就夠了？

因為 #009 的分層設計已經正確：

- `db/schema.ts` — DB 層，天生獨立
- `store/pg/PgStore.ts` — import `contracts.ts` 的型別，改成 Zod 後 `z.infer` 型別語意不變
- `auth/pg/PgAuth.ts` — 同上，`SessionUser` 型別語意不變

**介面設計正確時，改工具不需要改每一層。** 這是分層架構的核心價值體現。

- 日期：2026-04-26
- 分支：`feat/v8-clean-drizzle-neon-v2`
- 背景：在 `STORE_DRIVER=postgres`、`PG_SCHEMA=bf_v8` 下，發現切換帳號後偶發「加入購物車失敗」。

### 問題現象（案例）

1. 使用者 A（demo）登入後建立了 pending order。
2. 接著切換使用者 B（amy）操作加入購物車。
3. 前端偶發送出指向舊 orderId 的更新請求，後端回 `403 ORDER_NOT_OWNED` 或 `404`。

### 檢查結論（當前實作）

1. 登出流程有執行狀態清除（`setUser(null)`、清 localStorage、重置購物車）。
2. 但前端存在「非同步請求晚到覆寫」風險：舊使用者的訂單查詢結果可能在切換後才回來，造成短暫狀態污染。
3. 目前已加入保護：`addToCart` 在 `403/404` 時清除 `orderId`、重新載入 current order 後重試一次。

### 為什麼這一輪不做更多修補

本次決策是「不再擴大臨時修補」，原因如下：

1. 下一步即將導入 Better Auth，登入與 session 來源本來就會重構。
2. 現在加做前端過渡邏輯，極可能在下一階段被整段替換，重工成本高。
3. 目前有可用防護可維持操作穩定，足以支撐進入下一里程碑。

### 教學決策（關鍵訊息）

> 這次事件不是要證明「再補幾個 if 就能長治久安」，而是反過來證明：
> 當系統仍依賴前端保存 user/session 狀態時，狀態一致性與權限邊界會持續脆弱。
> 因此導入 Better Auth 不是功能加值，而是架構必要。

### 對 V9（Better Auth）產生的明確需求

1. Session 必須以 server 驗證結果為單一真相來源（single source of truth）。
2. 訂單 API 不再信任前端傳入 `userId`，改以 session user 判定歸屬。
3. 登入、登出、session 失效都走統一清理路徑，避免殘留狀態分叉。
4. 前端只做顯示狀態，不承擔「宣告我是誰」的責任。

### 結論

此紀錄正式將「登入/登出狀態競態」定義為 V9 要解決的架構問題，並作為導入 Better Auth 的第一性理由之一。
