## 背景與需求

### 專案現況

- **V9 版本**：已導入 Better Auth + Google OAuth
- **現有功能**：菜單管理、訂單系統、使用者認證
- **架構模式**：三層架構（contracts → route-schemas → backend）
- **資料庫**：PostgreSQL (Neon) + Drizzle ORM

### 業務需求

早餐店系統需要區分不同使用者的操作權限，確保：

- 顧客只能查看菜單、下單、查看自己的訂單
- 店員能協助顧客操作
- 廚師能查看待處理訂單
- 店長/老闆能管理菜單和查看所有訂單
- 系統管理員擁有完整控制權

## 架構哲學：事實發現與依賴邏輯

### 為什麼需要討論架構哲學？

在 AI 輔助開發的時代，我們面臨兩個極端：

**❌ Vibe Coding（憑感覺寫程式）**

```typescript
// 看到什麼寫什麼，沒有整體規劃
"幫我寫一個登入 API"
"再寫一個訂單 API"
"咦，訂單需要 userId，回頭改登入..."
"咦，角色權限怎麼處理？再改一次..."
→ 結果：混亂、重複、維護困難
```

**✅ Spec-Driven Development（規格驅動開發）**

```typescript
// 先定義規格，再實作
1. 定義 API contract
2. 根據 contract 生成程式碼
3. 確保前後端一致
→ 但缺少了「如何產生規格」的指引
```

**問題**：Spec-Driven Development 只解決了「有規格後怎麼做」，但沒有回答：

- 規格從哪裡來？
- 如何確保規格是「對的」？
- 如何避免規格本身就混亂？

### 🎯 事實發現與依賴邏輯：一個清晰的思維框架

#### 核心思想

軟體系統的混亂源於：**不清楚什麼是「事實」，什麼是「依賴事實產生的設計」**。

**解決方案**：建立明確的「事實層級」與「依賴方向」。

```
第1事實（領域真理）
    ↓ 依賴
第2事實（操作規格）
    ↓ 依賴
第3事實（實作細節）
```

**關鍵原則**：

- 下層依賴上層（不可反向）
- 每層都是上層的「具體化」
- 改動應該「向下傳播」，不應「向上修正」

### 📐 第1事實：業務領域的客觀真理

#### 定義

**第1事實是「外部世界的客觀存在」**，不是我們「設計」出來的，而是我們「發現」的。

```typescript
// 第1事實：早餐店的客觀事實
- 菜單項目（MenuItem）存在
- 訂單（Order）存在
- 顧客（Customer）存在
- 訂單包含多個菜單項目（OrderItem）

// 這些是「客觀存在」的概念，不是我們創造的
```

#### 在本專案中的體現：`shared/contracts.ts`

```typescript
// shared/contracts.ts - 第1事實的定義

// 菜單項目：客觀存在的「商品」
export const menuItemSchema = z.object({
  id: z.number(),
  name: z.string(), // 有名稱
  price: z.number(), // 有價格
  category: z.string(), // 屬於某分類
  description: z.string(),
  image_url: z.string(),
});

// 訂單項目：客觀存在的「訂購明細」
export const orderItemSchema = z.object({
  item: menuItemSchema, // 包含完整的菜單項目資訊
  qty: z.number().min(0), // 數量
});

// 訂單：客觀存在的「購買紀錄」
export const orderSchema = z.object({
  id: z.number(),
  userId: z.string(), // 由某人下單
  items: z.array(orderItemSchema), // 包含多個項目
  total: z.number(), // 有總金額
  status: z.enum(["pending", "submitted"]), // 有狀態
  createdAt: z.string(),
  submittedAt: z.string().optional(),
});
```

#### 第1事實的特徵

✅ **獨立於系統實作**

- 不管用 PostgreSQL 還是 MongoDB，菜單項目都有「名稱」和「價格」
- 不管用 REST 還是 GraphQL，訂單都包含「項目」和「總金額」

✅ **反映業務本質**

- 來自於對業務的觀察與理解
- 用業務語言描述（不是技術術語）

✅ **穩定性高**

- 除非業務本質改變，否則不會變
- 例如：「菜單項目有價格」這個事實不會因為技術選型而改變

✅ **唯一性**

- 每個概念只有一個定義（Single Source of Truth）
- 所有其他層都必須遵循這個定義

### 🔧 第2事實：操作流程與 API 規格

#### 定義

**第2事實是「如何操作第1事實」的設計**，這是我們根據業務需求「創造」的。

```typescript
// 第2事實：如何操作這些概念
- 如何「查詢」菜單？ → GET /api/menu
- 如何「建立」訂單？ → POST /api/orders
- 如何「更新」訂單狀態？ → PATCH /api/orders/:id

// 這些是「設計」出來的操作流程
```

#### 在本專案中的體現：`shared/route-schemas.ts`

```typescript
// shared/route-schemas.ts - 第2事實的定義

// 操作：建立菜單項目（需要哪些資料？）
export const createMenuItemBodySchema = z.object({
  name: z.string().min(1),
  price: z.number().min(0),
  category: z.string().min(1),
  description: z.string(),
  image_url: z.string().min(1),
});
// ↑ 依賴第1事實（menuItemSchema），但去掉了 id（由系統生成）

// 操作：更新訂單狀態（可以改什麼？）
export const updateOrderBodySchema = z.object({
  status: z.enum(["pending", "submitted"]),
});
// ↑ 依賴第1事實（orderSchema），但只允許改 status

// 操作：查詢訂單（回傳什麼格式？）
export const orderResponseEnvelopeSchema = z.object({
  data: orderSchema.nullable(), // ← 依賴第1事實
});
```

#### 第2事實的特徵

✅ **依賴第1事實**

- 所有 API 的 input/output 都基於第1事實的型別
- 不能創造「第1事實沒有的概念」

✅ **體現業務流程**

- 反映「使用者如何與系統互動」
- 考慮權限、驗證、錯誤處理

✅ **設計空間**

- 同樣的第1事實，可以有不同的第2事實設計
- 例如：CRUD vs CQRS vs Event Sourcing

### 🔐 認證與授權：屬於第2事實

#### 關鍵爭議

**問題**：認證（Authentication）與授權（Authorization）應該放在哪一層？

**兩派觀點**：

##### 觀點 A：屬於第3層（系統架構）

```
理由：
- 認證是「技術細節」（JWT、Session、OAuth）
- 授權是「橫切關注點」（Cross-cutting Concern）
- 應該用 middleware / decorator 實作

架構：
領域層（業務邏輯）
  ↓
應用層（API）
  ↓
基礎設施層（認證、資料庫、快取）← 放這裡
```

##### 觀點 B：屬於第2層（業務邏輯）✅ **本專案採用**

```
理由：
- 認證與授權「直接影響業務流程」
- 不同角色看到「不同的業務規則」
- 權限是「業務約束」，不只是「技術實作」

架構：
第1事實：菜單、訂單、使用者
  ↓
第2事實：API 規格 + 認證 + 授權 ← 放這裡
  ↓
第3事實：資料庫、快取、部署
```

---

#### 為什麼本專案選擇「觀點 B」？

##### 理由 1：權限影響業務語意

```typescript
// 同一個 API，不同角色看到不同結果
GET /api/orders

// Customer 角色
→ 回傳：只有自己的訂單
→ 這是業務規則，不是技術細節

// Staff 角色
→ 回傳：所有人的訂單
→ 業務規則改變了

// 權限不只是「能不能呼叫」，而是「呼叫後看到什麼」
```

##### 理由 2：角色是第1事實的一部分

```typescript
// shared/contracts.ts
export const sessionUserSchema = z.object({
  id: z.string(),
  email: z.string(),
  name: z.string(),
  roles: z.array(roleSchema), // ← 角色是「使用者的屬性」
});

// 使用者「有角色」是客觀事實
// 就像菜單項目「有價格」一樣
```

##### 理由 3：權限檢查是 API 規格的一部分

```typescript
// shared/route-schemas.ts

// API 規格應該明確說明「誰可以呼叫」
POST /api/menu
  ↑ 需要 owner 或 admin 角色
  ↑ 這是 API 的「前置條件」（precondition）
  ↑ 就像「price 必須大於 0」一樣，是業務規則

// 如果權限在第3層，API 規格就不完整了
```

---

#### 對比：兩種觀點的實作差異

##### 觀點 A：權限在基礎設施層

```typescript
// backend.ts（觀點 A）
app.post(
  "/api/menu",
  async ({ body }) => {
    // 業務邏輯：建立菜單項目
    const item = await store.createMenuItem(body);
    return { data: item };
  },
  {
    body: createMenuItemBodySchema,
    // ← 權限不在這裡，而是在 middleware
  },
);

// middleware/auth.ts（觀點 A）
app.use("/api/menu", authMiddleware({ roles: ["owner", "admin"] }));

// 問題：
// 1. API 規格與權限分離，難以理解完整邏輯
// 2. middleware 順序錯誤會導致安全漏洞
// 3. 權限規則散落在不同檔案
```

##### 觀點 B：權限在業務邏輯層 ✅

```typescript
// backend.ts（觀點 B）
app.post(
  "/api/menu",
  async ({ request, body, set }) => {
    // 1. 認證：誰在操作？
    const user = await requireUser(request);

    // 2. 授權：這個人可以做這件事嗎？
    requireAnyRole(user, ["owner", "admin"]);

    // 3. 業務邏輯：執行操作
    const item = await store.createMenuItem(body);
    set.status = 201;
    return { data: item };
  },
  {
    body: createMenuItemBodySchema,
    detail: {
      summary: "Create menu item (owner/admin only)", // ← 權限在規格中
    },
    response: {
      201: menuItemResponseSchema,
      401: apiErrorResponseSchema, // ← 認證失敗
      403: apiErrorResponseSchema, // ← 授權失敗
    },
  },
);

// 優點：
// 1. API 規格完整（包含權限要求）
// 2. 權限檢查在業務邏輯中明確可見
// 3. 容易理解「誰可以做什麼」
```

### 🎯 重要設計決策：訂單項目的快照設計（Snapshot Pattern）

#### 業務問題：訂單是與客戶的法律契約

**核心問題**：當顧客下訂單後，菜單項目的資訊可能會變更：

- 🏷️ **價格調整**：「培根蛋餅」從 45 元漲到 50 元
- ✏️ **名稱修改**：「起司蛋餅」改名為「黃金起司蛋餅」
- 📝 **描述更新**：加入「使用法國進口起司」的說明
- 🖼️ **圖片更換**：換成更吸引人的商品照片
- 🗑️ **品項下架**：某個品項停售並從菜單中刪除

**如果訂單只存 `itemId`，會發生什麼？**

```typescript
// ❌ 錯誤設計：只存 itemId
const orderItems = [
  { orderId: 101, itemId: 5, qty: 2 }, // 只存 ID
];

// 問題1：價格變更後，歷史訂單金額對不上
const historicalOrder = getOrder(101);
const currentPrice = getMenuItem(5).price; // 取得現在的價格
const calculatedTotal = historicalOrder.items.reduce(
  (sum, item) => sum + getMenuItem(item.itemId).price * item.qty, // ← 錯誤！
  0,
);
// → 訂單總金額從 90 元變成 100 元（因為單價從 45 漲到 50）

// 問題2：品項刪除後，訂單無法顯示
const menuItem = getMenuItem(5); // → null（已刪除）
// → 無法顯示「這筆訂單買了什麼」
```

#### ✅ 正確設計：完整快照（Snapshot All Fields）

**V10 採用的設計：將訂單成立時的完整菜單資訊存入 order_items 表**

```typescript
// db/schema.ts
export const orderItemsTable = appSchema.table("order_items", {
  id: integer("id").primaryKey().generatedByDefaultAsIdentity(),
  orderId: integer("order_id")
    .notNull()
    .references(() => ordersTable.id, { onDelete: "cascade" }),

  // ✅ 保留原始 itemId（用於追溯與分析）
  itemId: integer("item_id").notNull(),

  // ✅ 完整快照當時的菜單資訊（與客戶的契約）
  name: text("name").notNull(), // 快照：品名
  price: integer("price").notNull(), // 快照：價格
  category: text("category").notNull(), // 快照：分類
  description: text("description").notNull(), // 快照：描述
  imageUrl: text("image_url").notNull(), // 快照：圖片URL

  qty: integer("qty").notNull(),
});
```

#### 📊 設計方案對比

| 設計方案                       | 優點                                                                                        | 缺點                                                                      | 適用場景                       |
| ------------------------------ | ------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------- | ------------------------------ |
| **方案1：只存 `itemId`**       | • 節省空間<br>• 菜單更新會同步反映                                                          | • 價格變更影響歷史訂單<br>• 品項刪除後訂單無法顯示<br>• 違反會計/法律要求 | ❌ 不適用於訂單                |
| **方案2：存 `itemId + price`** | • 保留歷史價格<br>• 節省部分空間                                                            | • 品名/描述變更影響顯示<br>• 品項刪除仍有問題<br>• 無法完整還原當時情況   | ⚠️ 僅適用於簡單場景            |
| **方案3：完整快照** ✅         | • 訂單永不受菜單變更影響<br>• 完整保留交易證據<br>• 符合會計/法律要求<br>• 品項刪除也不影響 | • 佔用較多空間<br>• 需要反正規化設計                                      | ✅ **電商/訂單系統的標準做法** |

#### 🔍 itemId 的作用：追溯與分析

**為什麼既要快照，又要保留 `itemId`？**

```typescript
// 1️⃣ 追溯：查詢「這個訂單項目對應哪個菜單？」
const orderItem = await db
  .select()
  .from(orderItemsTable)
  .where(eq(orderItemsTable.id, 123));

console.log(`訂購的是：${orderItem.name}（當時價格：${orderItem.price}）`);
console.log(`對應菜單 ID：${orderItem.itemId}`);

// 2️⃣ 分析：哪個菜單項目最暢銷？
const topSelling = await db
  .select({
    itemId: orderItemsTable.itemId,
    itemName: orderItemsTable.name, // 用快照的名稱（避免改名影響統計）
    totalQty: sql<number>`sum(${orderItemsTable.qty})`,
  })
  .from(orderItemsTable)
  .groupBy(orderItemsTable.itemId, orderItemsTable.name)
  .orderBy(desc(sql`sum(${orderItemsTable.qty})`));

// 3️⃣ 對比：價格變化對銷量的影響
const priceHistory = await db
  .select({
    itemId: orderItemsTable.itemId,
    orderDate: ordersTable.createdAt,
    priceAtOrder: orderItemsTable.price, // 快照價格
    currentPrice: menuItemsTable.price, // 現在價格
  })
  .from(orderItemsTable)
  .innerJoin(ordersTable, eq(orderItemsTable.orderId, ordersTable.id))
  .leftJoin(menuItemsTable, eq(orderItemsTable.itemId, menuItemsTable.id))
  .where(eq(orderItemsTable.itemId, 5));
```

#### 💡 實作考量

**1. 反正規化（Denormalization）是必要的**

- 訂單表違反了資料庫正規化原則（重複存儲菜單資訊）
- 但這是**有意為之**，因為業務需求優先於技術範式

**2. 快照時機：訂單提交時（不是加入購物車時）**

```typescript
// backend.ts - 提交訂單時快照
app.post("/api/orders/:id/submit", async ({ params, db }) => {
  const orderId = parseInt(params.id);

  // 1. 從購物車取得 itemIds
  const cartItems = await getCartItems(orderId);

  // 2. 查詢當前菜單資訊
  const menuItems = await db
    .select()
    .from(menuItemsTable)
    .where(
      inArray(
        menuItemsTable.id,
        cartItems.map((i) => i.itemId),
      ),
    );

  // 3. 快照完整資訊到 order_items
  await db.insert(orderItemsTable).values(
    cartItems.map((cart) => {
      const menu = menuItems.find((m) => m.id === cart.itemId)!;
      return {
        orderId,
        itemId: menu.id, // 保留 ID
        name: menu.name, // 快照
        price: menu.price, // 快照
        category: menu.category, // 快照
        description: menu.description, // 快照
        imageUrl: menu.imageUrl, // 快照
        qty: cart.qty,
      };
    }),
  );

  // 4. 更新訂單狀態
  await db
    .update(ordersTable)
    .set({ status: "submitted", submittedAt: new Date() })
    .where(eq(ordersTable.id, orderId));
});
```

## 🔍 進階討論：訂單項目設計的兩種架構方案

### 📌 設計問題的本質

在前面的快照設計說明中，我們展示了 V10 如何在 `order_items` 表中儲存完整的菜單資訊（名稱、價格、描述、圖片等）。但這引發了一個深層的架構問題：

**核心問題**：當菜單項目可能變更（改價、改名、下架）時，訂單應該如何引用菜單資訊？

這個問題有兩種根本不同的解決方案：

1. **方案A：業務層版本化（Immutable Menu Items）** - 菜單項目本身就是不可變的，「修改」= 創建新版本
2. **方案B：實作層快照（Snapshot on Order）** - 菜單項目可變，但在訂單提交時做完整快照

---

### 方案A：業務層版本化（Business-Level Immutability）

#### 核心概念

**菜單項目是不可變實體（Immutable Entity），每次「修改」都會創建新版本**

```typescript
// 業務模型：菜單項目有版本概念
type MenuItem = {
  id: string; // "001-01", "001-02" （物理ID，含版本號）
  entityId: string; // "e7f6d5c4-..." （邏輯ID，永不改變）
  logicalId: string; // "001" （業務ID，用於分組）
  version: number; // 1, 2, 3...

  // 業務資料
  name: string;
  price: number;
  category: string;
  description: string;
  imageUrl: string;

  // 版本管理
  isCurrentVersion: boolean;
  supersedes: string | null; // 取代哪個版本
  changeReason: string; // 為什麼改版
  createdAt: Date;
  createdBy: string;
};
```

#### 資料庫設計


#### 業務操作流程

```typescript
// 1️⃣ 顯示當前菜單（前端）
async function getCurrentMenu() {
  const items = await db
    .select()
    .from(menuItemsTable)
    .where(eq(menuItemsTable.isCurrentVersion, true))
    .orderBy(menuItemsTable.logicalId);

  return items; // 只顯示最新版本
}

// 2️⃣ 修改菜單（後台）- 創建新版本
async function updateMenuItem(
  logicalId: string,
  changes: Partial<MenuItem>,
  reason: string,
) {
  return await db.transaction(async (tx) => {
    // Step 1: 找到當前版本
    const currentVersion = await tx
      .select()
      .from(menuItemsTable)
      .where(
        and(
          eq(menuItemsTable.logicalId, logicalId),
          eq(menuItemsTable.isCurrentVersion, true),
        ),
      )
      .limit(1);

    if (!currentVersion[0]) throw new Error("Menu item not found");

    // Step 2: 標記舊版本為非當前
    await tx
      .update(menuItemsTable)
      .set({ isCurrentVersion: false })
      .where(eq(menuItemsTable.id, currentVersion[0].id));

    // Step 3: 創建新版本
    const newVersion = currentVersion[0].version + 1;
    const newId = `${logicalId}-${String(newVersion).padStart(2, "0")}`;

    await tx.insert(menuItemsTable).values({
      id: newId,
      entityId: currentVersion[0].entityId,
      logicalId: logicalId,
      version: newVersion,
      name: changes.name ?? currentVersion[0].name,
      price: changes.price ?? currentVersion[0].price,
      category: changes.category ?? currentVersion[0].category,
      description: changes.description ?? currentVersion[0].description,
      imageUrl: changes.imageUrl ?? currentVersion[0].imageUrl,
      isCurrentVersion: true,
      supersedes: currentVersion[0].id,
      changeReason: reason,
      createdBy: getCurrentUserId(),
    });

    return newId;
  });
}

// 3️⃣ 提交訂單（只存 menuItemId）
async function submitOrder(orderId: number) {
  const cartItems = await getCartItems(orderId);

  // 驗證：確保引用的是當前版本
  const menuItemIds = cartItems.map((item) => item.menuItemId);
  const menuItems = await db
    .select()
    .from(menuItemsTable)
    .where(inArray(menuItemsTable.id, menuItemIds));

  // ⚠️ 可選：檢查是否有非當前版本（購物車過期）
  const outdatedItems = menuItems.filter((item) => !item.isCurrentVersion);
  if (outdatedItems.length > 0) {
    throw new Error("購物車中有項目已更新，請重新確認");
  }

  // 插入訂單項目（簡單！）
  await db.insert(orderItemsTable).values(
    cartItems.map((item) => ({
      orderId,
      menuItemId: item.menuItemId, // ← 只存 ID
      qty: item.qty,
    })),
  );

  // 更新訂單狀態
  await db
    .update(ordersTable)
    .set({ status: "submitted", submittedAt: new Date() })
    .where(eq(ordersTable.id, orderId));
}

// 4️⃣ 查詢歷史訂單（JOIN 取得完整資訊）
async function getOrderDetails(orderId: number) {
  const result = await db
    .select({
      orderId: ordersTable.id,
      orderStatus: ordersTable.status,
      orderTotal: ordersTable.total,
      itemId: orderItemsTable.id,
      itemName: menuItemsTable.name,
      itemPrice: menuItemsTable.price,
      itemDescription: menuItemsTable.description,
      itemImageUrl: menuItemsTable.imageUrl,
      itemQty: orderItemsTable.qty,
      itemVersion: menuItemsTable.version,
    })
    .from(ordersTable)
    .innerJoin(orderItemsTable, eq(orderItemsTable.orderId, ordersTable.id))
    .innerJoin(
      menuItemsTable,
      eq(orderItemsTable.menuItemId, menuItemsTable.id),
    )
    .where(eq(ordersTable.id, orderId));

  return result; // ← 自動取得歷史版本的正確資訊
}
```

### 方案B：實作層快照（Implementation-Level Snapshot）

#### 核心概念

**菜單項目是可變實體（Mutable Entity），訂單提交時做完整快照**

這就是 V10 目前的實作方式。詳細說明請參考前面的「🎯 重要設計決策：訂單項目的快照設計」章節。

#### 快速回顧

### 📊 兩種方案的全面對比

#### 1. 架構層面

| 維度               | 方案A：業務層版本化      | 方案B：實作層快照                  |
| ------------------ | ------------------------ | ---------------------------------- |
| **設計哲學**       | Immutability（不可變性） | Mutation + Snapshot（可變 + 快照） |
| **複雜度位置**     | 業務邏輯層               | 資料儲存層                         |
| **versioning**     | 顯式版本管理（Explicit） | 隱式快照（Implicit）               |
| **Event Sourcing** | 天然支援                 | 不支援                             |
| **CQRS 友好度**    | 高（讀寫分離）           | 中                                 |

#### 2. 資料儲存

| 維度             | 方案A                          | 方案B                |
| ---------------- | ------------------------------ | -------------------- |
| **菜單表大小**   | ⚠️ 較大（保留所有版本）        | ✅ 較小（一項一筆）  |
| **訂單表大小**   | ✅ 較小（只存ID）              | ⚠️ 較大（完整快照）  |
| **總體儲存成本** | 低\*（訂單遠多於菜單改版）     | 高（每筆訂單都複製） |
| **查詢複雜度**   | 需要 JOIN                      | 直接讀取             |
| **索引策略**     | 需要 `is_current_version` 索引 | 簡單索引             |

**儲存成本分析**：

```
假設：
- 100 個菜單項目，平均 5 次改版 = 500 筆（每筆 1KB = 500KB）
- 10,000 個訂單，平均 3 個項目 = 30,000 筆快照（每筆 1KB = 30MB）

結論：訂單量大時，方案B 的儲存成本是方案A 的 60 倍
```

#### 3. 業務能力

| 能力         | 方案A       | 方案B         | 說明                             |
| ------------ | ----------- | ------------- | -------------------------------- |
| **變更歷史** | ✅ 完整     | ❌ 無         | 方案A 可知道「何時」「為何」改版 |
| **審計追蹤** | ✅ 內建     | ⚠️ 需額外實作 | 方案A 符合 SOX、GDPR 等合規要求  |
| **回溯查詢** | ✅ 支援     | ❌ 不支援     | 「去年3月的菜單長什麼樣？」      |
| **影響分析** | ✅ 精確     | ⚠️ 間接       | 「調價後銷量如何變化？」         |
| **A/B 測試** | ✅ 天然支援 | ❌ 需額外設計 | 可同時存在多個「當前版本」       |
| **訂單保護** | ✅ 自動     | ✅ 自動       | 兩者都能保護歷史訂單             |

### 🎯 方案A 的「缺點」辨析

在決策過程中，方案A 的幾個「缺點」經常被提出。讓我們逐一分析：

#### ❌ D1: 菜單表會膨脹

**反駁** ✅ ：

對企業而言，**菜單變更本身就是重大商業決策**：

- 📊 **品牌影響**：改價、改名會影響品牌形象和顧客認知
- 💰 **營收影響**：價格調整直接影響銷量和利潤
- 📋 **合規要求**：許多行業需要保留價格變更記錄（如食品安全、消費者保護）
- 🎯 **策略決策**：需要分析歷史數據來支持未來決策

因此：

- 菜單變更頻率**遠低於**訂單生成頻率（通常是 1:1000 或更低）
- 即使 100 個菜單項目各改版 10 次 = 1,000 筆記錄 = 1MB（可忽略）
- 相比之下，10,000 筆訂單的快照 = 30MB（30 倍差異）

**結論**：在合理的業務場景下，菜單表不會大幅膨脹。

---

#### ❌ D2: 查詢複雜度提高

**反駁** ✅ ：

**效能角度**（實測無問題）：

```sql
-- 正確的索引設計
CREATE INDEX idx_menu_current ON menu_items(is_current_version)
WHERE is_current_version = true;

-- 查詢效能：O(n) where n = 當前版本數量（通常 < 100）
SELECT * FROM menu_items WHERE is_current_version = true;
```

即使有 10,000 筆歷史版本，上述查詢也能在毫秒內完成。

**架構角度**（可控的複雜度）：

```typescript
// 方法1：使用 Repository Pattern
class MenuRepository {
  async getCurrentMenu() {
    return db.select()
      .from(menuItemsTable)
      .where(eq(menuItemsTable.isCurrentVersion, true));
  }

  async getMenuHistory(logicalId: string) {
    return db.select()
      .from(menuItemsTable)
      .where(eq(menuItemsTable.logicalId, logicalId))
      .orderBy(desc(menuItemsTable.version));
  }
}

// 方法2：使用資料庫 View（推薦）
CREATE VIEW current_menu AS
SELECT * FROM menu_items WHERE is_current_version = true;

// 方法3：Drizzle Subquery
const currentMenuItems = db
  .select()
  .from(menuItemsTable)
  .where(eq(menuItemsTable.isCurrentVersion, true))
  .as('current_menu');
```

**結論**：複雜度可以通過架構約束來控制，效能不是問題。

---

#### ❌ D3: 前端需要理解版本

**反駁** ✅ ：

**這是業務需求，不是技術債務**。

當菜單發生變更時，前端必須處理以下 UX 場景（無論採用哪種方案）：

| UX 場景          | 前端需求               | 方案A                  | 方案B                 |
| ---------------- | ---------------------- | ---------------------- | --------------------- |
| **購物車失效**   | 價格變更通知           | ✅ 自動檢測            | ⚠️ 需手動實作         |
| **價格變動提示** | 顯示「最近調價」標籤   | ✅ `isNewVersion` 欄位 | ⚠️ 需比對 `updatedAt` |
| **歷史訂單顯示** | 顯示當時的價格         | ✅ 自動正確            | ✅ 快照正確           |
| **客服查詢**     | 「為什麼價格不一樣？」 | ✅ 有 `changeReason`   | ❌ 無法解釋           |

**前端實作範例**：

```typescript
// shared/contracts.ts - API 層抽象
export const menuItemSchema = z.object({
  id: z.string(),              // "001-02"（前端不需理解版本號）
  displayId: z.string(),       // "1號餐"（顯示用）
```

#### ❌ D4: 級聯更新問題

**反駁** ✅ ：

這個「缺點」實際上揭示了**ID 語意混淆**的設計問題。

**問題場景**：「1號餐要重新編號為3號餐」

**錯誤的理解**：需要修改所有版本的 `id` 欄位  
**正確的理解**：需要分離不同層次的 ID

```typescript
// db/repositories/menuRepository.ts
import { db } from "../index";
import { menuItemsTable } from "../schema";
import { eq, and, desc, sql } from "drizzle-orm";

export class MenuRepository {
  /**
   * 取得當前菜單（只包含最新版本）
   */
  async getCurrentMenu() {
    return await db
      .select()
      .from(menuItemsTable)
      .where(eq(menuItemsTable.isCurrentVersion, true))
      .orderBy(menuItemsTable.logicalId);
  }

  /**
   * 取得特定菜單項目的所有版本
   */
  async getMenuVersionHistory(logicalId: string) {
    return await db
      .select()
      .from(menuItemsTable)
      .where(eq(menuItemsTable.logicalId, logicalId))
      .orderBy(desc(menuItemsTable.version));
  }

  /**
   * 取得特定版本的菜單項目
   */
  async getMenuVersion(id: string) {
    const result = await db
      .select()
      .from(menuItemsTable)
      .where(eq(menuItemsTable.id, id))
      .limit(1);

    return result[0] ?? null;
  }

  /**
   * 創建新的菜單項目（version 1）
   */
  async createMenuItem(data: {
    logicalId: string;
    name: string;
    price: number;
    category: string;
    description: string;
    imageUrl: string;
    createdBy: string;
  }) {
    const entityId = crypto.randomUUID();
    const id = `${data.logicalId}-01`;

    await db.insert(menuItemsTable).values({
      id,
      entityId,
      logicalId: data.logicalId,
      version: 1,
      name: data.name,
      price: data.price,
      category: data.category,
      description: data.description,
      imageUrl: data.imageUrl,
      isCurrentVersion: true,
      changeReason: "Initial creation",
      createdBy: data.createdBy,
    });

    return id;
  }

  /**
   * 更新菜單項目（創建新版本）
   */
  async updateMenuItem(
    logicalId: string,
    changes: Partial<
      Pick<MenuItem, "name" | "price" | "category" | "description" | "imageUrl">
    >,
    reason: string,
    userId: string,
  ) {
    return await db.transaction(async (tx) => {
      // 1. 找到當前版本
      const currentVersion = await tx
        .select()
        .from(menuItemsTable)
        .where(
          and(
            eq(menuItemsTable.logicalId, logicalId),
            eq(menuItemsTable.isCurrentVersion, true),
          ),
        )
        .limit(1);

      if (!currentVersion[0]) {
        throw new Error(`Menu item with logicalId ${logicalId} not found`);
      }

      const current = currentVersion[0];

      // 2. 標記舊版本為非當前
      await tx
        .update(menuItemsTable)
        .set({ isCurrentVersion: false })
        .where(eq(menuItemsTable.id, current.id));

      // 3. 創建新版本
      const newVersion = current.version + 1;
      const newId = `${logicalId}-${String(newVersion).padStart(2, "0")}`;

      await tx.insert(menuItemsTable).values({
        id: newId,
        entityId: current.entityId,
        logicalId: logicalId,
        version: newVersion,
        name: changes.name ?? current.name,
        price: changes.price ?? current.price,
        category: changes.category ?? current.category,
        description: changes.description ?? current.description,
        imageUrl: changes.imageUrl ?? current.imageUrl,
        isCurrentVersion: true,
        supersedes: current.id,
        changeReason: reason,
        createdBy: userId,
      });

      return newId;
    });
  }

  /**
   * 檢查菜單項目是否為當前版本（用於購物車驗證）
   */
  async validateMenuItemsAreCurrentValidation(menuItemIds: string[]) {
    const items = await db
      .select({
        id: menuItemsTable.id,
        isCurrentVersion: menuItemsTable.isCurrentVersion,
      })
      .from(menuItemsTable)
      .where(sql`${menuItemsTable.id} = ANY(${menuItemIds})`);

    const outdated = items.filter((item) => !item.isCurrentVersion);

    return {
      valid: outdated.length === 0,
      outdatedIds: outdated.map((item) => item.id),
    };
  }
}

export const menuRepository = new MenuRepository();
```

```typescript
        .innerJoin(
          menuItemsTable,
          eq(orderItemsTable.menuItemId, menuItemsTable.id),
        )
        .innerJoin(ordersTable, eq(orderItemsTable.orderId, ordersTable.id))
        .where(
          and(
            eq(menuItemsTable.logicalId, logicalId),
            eq(menuItemsTable.version, period.version),
            gte(ordersTable.createdAt, period.effectiveFrom),
            period.effectiveTo
              ? lt(ordersTable.createdAt, period.effectiveTo)
              : sql`true`,
          ),
        );

      return {
        ...period,
        ...sales[0],
      };
    }),
  );

  return salesByPrice;
}
```

#### 3. 促銷管理系統（V12）

```typescript
// 基於版本的促銷系統
export const promotionsTable = appSchema.table("promotions", {
  id: integer("id").primaryKey().generatedByDefaultAsIdentity(),
  name: text("name").notNull(),
  targetType: text("target_type").notNull(), // "logical_id" | "specific_version"
  targetId: text("target_id").notNull(), // "001" 或 "001-02"
  discountType: text("discount_type").notNull(), // "percentage" | "fixed"
  discountValue: integer("discount_value").notNull(),
  startDate: timestamp("start_date").notNull(),
  endDate: timestamp("end_date").notNull(),
});

// 查詢邏輯
async function getMenuWithPromotions() {
  const now = new Date();

  const menu = await db
    .select({
      ...getTableColumns(menuItemsTable),
      promotionId: promotionsTable.id,
      promotionName: promotionsTable.name,
      discountType: promotionsTable.discountType,
      discountValue: promotionsTable.discountValue,
    })
    .from(menuItemsTable)
    .leftJoin(
      promotionsTable,
      and(
        or(
          and(
            eq(promotionsTable.targetType, "logical_id"),
            eq(promotionsTable.targetId, menuItemsTable.logicalId),
          ),
          and(
            eq(promotionsTable.targetType, "specific_version"),
            eq(promotionsTable.targetId, menuItemsTable.id),
          ),
        ),
        lte(promotionsTable.startDate, now),
        gte(promotionsTable.endDate, now),
      ),
    )
    .where(eq(menuItemsTable.isCurrentVersion, true));

  return menu;
}
```

### 🚀 這個架構如何改善 AI 協作？

#### 問題：Vibe Coding 為什麼失敗？

```
開發者：「幫我寫一個登入 API」
AI：「好的，這是 JWT 登入...」
開發者：「再寫一個訂單 API」
AI：「好的，這是訂單 CRUD...」
開發者：「咦，訂單需要知道 userId...」
開發者：「咦，權限怎麼處理...」
→ 不斷回頭修改，架構越來越亂
```

**根本問題**：沒有先定義「事實」，直接跳到「實作」。

---

#### 解決方案：分層提示（Layered Prompting）

##### Phase 1：發現第1事實

```
提示詞：
「我要開發一個早餐店訂餐系統。
請幫我分析業務領域，列出核心概念（entity）及其屬性。
不要考慮實作細節，只描述業務本質。」

AI 輸出：
- MenuItem（菜單項目）：name, price, category
- Order（訂單）：userId, items[], total, status
- User（使用者）：id, email, name
- OrderItem（訂單項目）：menuItem, quantity

開發者審查：
「這些是客觀存在的概念嗎？」
「有遺漏嗎？」
→ 確認後，寫成 contracts.ts
```

##### Phase 2：設計第2事實

```
提示詞：
「根據以下的業務概念（附上 contracts.ts），
設計 RESTful API 規格。考慮：
1. CRUD 操作
2. 權限控制（Customer, Staff, Chef, Owner, Admin）
3. 輸入驗證
4. 錯誤處理」

AI 輸出：
GET    /api/menu          → 所有人可存取
POST   /api/menu          → 需要 owner/admin
PATCH  /api/menu/:id      → 需要 owner/admin
DELETE /api/menu/:id      → 需要 owner/admin

GET    /api/orders        → 根據角色過濾
POST   /api/orders        → 需要登入
PATCH  /api/orders/:id    → 需要 chef/owner/admin

開發者審查：
「這些操作符合業務流程嗎？」
「權限設計合理嗎？」
→ 確認後，寫成 route-schemas.ts
```

##### Phase 3：實作第3事實

```
提示詞：
「根據以下的 API 規格（附上 route-schemas.ts），
用 Drizzle ORM 設計 PostgreSQL schema。考慮：
1. 正規化（避免資料重複）
2. 外鍵約束（確保資料完整性）
3. 索引優化（常見查詢）」

AI 輸出：
CREATE TABLE menu_items (...)
CREATE TABLE orders (...)
CREATE TABLE order_items (...)
CREATE INDEX idx_orders_user_id ON orders(user_id);

開發者審查：
「這個 schema 能支援所有 API 操作嗎？」
「有效能瓶頸嗎？」
→ 確認後，寫成 db/schema.ts
```

#### 關鍵優勢

**✅ 清晰的溝通**

- 每次對話專注在「一個層級」
- AI 不會混淆「業務概念」和「實作細節」

**✅ 可驗證性**

- 每層都可以獨立審查
- 發現問題可以「向下修正」，不用「向上打補丁」

**✅ 可維護性**

- 業務需求變更？改第1事實
- API 設計調整？改第2事實
- 效能優化？改第3事實
- 改動範圍明確，影響可控

**✅ 團隊協作**

- 業務分析師：定義第1事實
- API 設計師：定義第2事實
- 後端工程師：實作第3事實
- 分工清晰，減少衝突

### 🎓 本專案的實踐

#### 為什麼這個專案叫「Contract-First」？

```
傳統開發流程：
資料庫 → 後端 API → 前端
↑ 問題：資料庫 schema 主導設計，業務邏輯被技術細節綁架

Contract-First 流程：
第1事實（contracts.ts）
  ↓
第2事實（route-schemas.ts + auth）
  ↓
第3事實（db/schema.ts）
↑ 優勢：業務邏輯主導設計，技術細節服務業務
```

#### 實際案例：RBAC 的設計過程

**Step 1：發現第1事實**

```typescript
// 觀察業務：早餐店有不同的「角色」
export const roleSchema = z.enum([
  "admin",
  "owner",
  "chef",
  "staff",
  "customer",
]);

// 使用者「有」角色（客觀事實）
export const sessionUserSchema = z.object({
  id: z.string(),
  email: z.string(),
  name: z.string(),
  roles: z.array(roleSchema), // ← 新增
});
```

**Step 2：設計第2事實**

```typescript
// 設計 API 規格：誰可以做什麼？
POST /api/menu → requireAnyRole(["owner", "admin"])
PATCH /api/orders/:id → requireAnyRole(["chef", "owner", "admin"])
GET /api/orders →
  - customer: 只看自己的
  - staff+: 看所有的
```

**Step 3：實作第3事實**

```typescript
// 資料庫層：如何儲存角色？
export const user = appSchema.table("user", {
  // ...
  roles: text("roles").array().notNull().default(["customer"]),

// 選擇 PostgreSQL array 而非 many-to-many（效能考量）
```

**結果**：

- 業務邏輯清晰（第1、2事實）
- 實作細節隱藏（第3事實）
- 未來可以換成 many-to-many（只改第3事實）

### 💡 與傳統架構模式的對比

#### MVC（Model-View-Controller）

```
Model     ← 資料庫邏輯
Controller ← 業務邏輯
View      ← 前端

問題：
- Model 通常直接對應資料庫 table（第3事實主導）
- 缺少明確的「業務概念層」（第1事實）
```

#### 三層架構（Presentation-Business-Data）

```
Presentation ← 前端
Business     ← 業務邏輯
Data         ← 資料庫

問題：
- Business 層太模糊（什麼都可以放）
- 沒有區分「業務概念」和「業務流程」
```

#### DDD（Domain-Driven Design）

```
Domain Layer    ← Entity, Value Object
Application Layer ← Use Case
Infrastructure Layer ← Database, API

類似於：
Domain ≈ 第1事實
Application ≈ 第2事實
Infrastructure ≈ 第3事實

優點：最接近我們的思維
缺點：學習曲線陡峭，概念複雜
```

#### 本專案的「事實層級」架構

```
第1事實（contracts.ts）     ← 清晰、簡單
第2事實（route-schemas.ts）  ← 實用、直接
第3事實（db/schema.ts）      ← 彈性、可替換

優點：
- 比 MVC 更清晰（業務概念明確）
- 比三層架構更具體（職責明確）
- 比 DDD 更簡單（學習曲線平緩）
- 適合教學與中小型專案
```

### ✨ 總結：架構哲學的核心原則

#### 1. 事實優先於設計

```
❌ 錯誤：「我們用 PostgreSQL，所以要設計正規化 schema」
✅ 正確：「業務有這些概念，用 PostgreSQL 如何表達？」

第3事實服務第1事實，不是反過來
```

#### 2. 業務語言優先於技術術語

```
❌ 錯誤：「這個 entity 有 foreign key 指向 user table」
✅ 正確：「訂單屬於某個使用者」

第1事實用業務語言，第3事實用技術語言
```

#### 3. 穩定層不依賴變動層

```
第1事實（最穩定）
  ↓
第2事實（中等穩定）
  ↓
第3事實（最易變）

依賴方向永遠向上（穩定的方向）
```

#### 4. 認證授權是業務邏輯

```
權限不是「加裝的安全機制」
而是「業務規則的一部分」

「顧客只能看自己的訂單」
↑ 這是業務規則，不是技術細節
```

#### 5. AI 協作需要分層思考

```
Phase 1: 讓 AI 幫你「發現事實」
Phase 2: 讓 AI 幫你「設計流程」
Phase 3: 讓 AI 幫你「優化實作」

不要一次要求 AI 做所有事
```

## 版本管理策略

### V10 的定位：快照式演化（Snapshot Evolution）

V10 是從 V9 **clone** 而來的獨立版本，而非透過 git merge 或繼承方式演化。

```
v7 ─clone→ v8 ─clone→ v9 ─clone→ v10
 ↓          ↓          ↓          ↓
穩定       穩定       穩定       獨立演化
(凍結)     (凍結)     (凍結)     (active)
```

**核心原則**：

- ✅ Clone 時複製完整的程式碼與架構
- ✅ Clone 後完全獨立，不再與來源版本連動
- ✅ 每個版本都有獨立的資料庫 schema（`bf_v7` / `bf_v8` / `bf_v9` / `bf_v10`）
- ✅ 舊版本凍結，作為穩定的學習參考

---

### 為什麼這樣做？

#### 教學場景 vs 產品開發

| 維度       | 教學場景（本專案策略） | 產品開發（傳統策略） |
| ---------- | ---------------------- | -------------------- |
| 版本關係   | 獨立快照（clone）      | 持續演化（merge）    |
| 資料庫     | 隔離（v7/v8/v9/v10）   | 共用 + migration     |
| Bug 修復   | 各版本獨立處理         | 統一修復後 backport  |
| 程式碼重用 | 低（允許重複）         | 高（DRY 原則）       |
| 穩定性     | 極高（舊版凍結）       | 中（持續變動）       |
| 學習曲線   | 平緩（可對照）         | 陡峭（只看最新）     |
| 適用場景   | 📚 教學、研究          | 🚀 商業產品          |

---

### 這個策略的好處

#### 1. 教學導向

- 學生可以**同時運行多個版本**對照學習

  ```bash
  # Terminal 1
  cd v9 && bun run dev  # port 3000

  # Terminal 2
  cd v10 && bun run dev # port 3001
  ```

- 清晰的**迭代軌跡**：從簡單（v7）到複雜（v10）
- 可以**隨時回退**查看特定功能的實作方式

#### 2. 穩定性保證

- V9 的後續改動**不會破壞** V10 的實驗
- 每個版本都是**可運行的完整範例**
- 不會因為依賴更新而導致舊版本失效

#### 3. 獨立實驗空間

- V10 可以**大膽重構** RBAC，不擔心影響 V9
- 如果實驗失敗，V9 仍是穩定的 fallback
- 資料庫隔離，測試**不會互相干擾**

#### 4. 版本對比容易

```bash
# 直接 diff 比較檔案
diff v9/backend.ts v10/backend.ts

# 比較 schema
diff v9/db/auth-schema.ts v10/db/auth-schema.ts

# 比較資料庫結構
SELECT table_name FROM information_schema.tables
WHERE table_schema = 'bf_v9';

SELECT table_name FROM information_schema.tables
WHERE table_schema = 'bf_v10';
```

---

### 這個策略的代價

#### 1. 維護成本較高

- Bug 修復需要**手動同步**到多個版本（如果需要）
- 共用邏輯（如 `util.ts`）無法統一更新
- 違反 **DRY 原則**（Don't Repeat Yourself）

#### 2. 程式碼重複

- 每個版本都有完整的 codebase
- 磁碟空間佔用較大
- 相似的程式碼片段散落各處

#### 3. 升級路徑需手動處理

- 學生想從 V9 升級到 V10，需要**手動遷移**
- 沒有自動化的 upgrade script
- 需要詳細的升級文檔（如本講義）

⚠️ **權衡**：對於教學場景，**穩定性 > 維護效率**，因此這個策略是合理的。

#### 1. 資料庫完全獨立

### 深入探討：為什麼不用 Git Merge？

#### Git 真的能做到「繼承」嗎？

**問題**：如果用 Git 分支管理版本（v7/v8/v9/v10），能讓「v9 改了，v10 自動跟著改」嗎？

**答案**：技術上可以，但會造成嚴重混亂。

---

#### 方案對比：Merge 策略 vs Clone 策略

##### 方案 A：Merge 策略（理論上的「繼承」）

```bash
# 分支結構
main
  ├── feat/v7
  ├── feat/v8 (從 v7 分支)
  ├── feat/v9 (從 v8 分支)
  └── feat/v10 (從 v9 分支)

# V9 修正了一個 bug
cd feat/v9
git commit -m "fix: 修正 session 過期問題"

# V10 想要這個修正
cd feat/v10
git merge feat/v9  # ← 這裡就會出問題！
```

---

#### 🔥 Merge 策略的四大問題

##### 問題 1：無法選擇性繼承

```
v9 的改動：
  ✅ bug fix: session 過期邏輯
  ❌ new feature: 新增 email 通知
  ❌ refactor: 重構 auth 流程

v10 想要：
  ✅ bug fix（要）
  ❌ new feature（不要，v10 有自己的通知系統）
  ❌ refactor（不要，v10 的 auth 已經改成 RBAC）

問題：git merge 是「全拿或全不拿」，無法挑選！
```

**現實場景**：

- V9 加了一個新功能（如 email 驗證）
- V10 不需要這個功能（用 Google OAuth）
- 但 V9 同時也修了一個重要 bug
- 如果 merge，你會**被迫接受**所有改動，包括不想要的

#### ⚖️ 物件導向的繼承 vs Git 的繼承

```typescript
// ✅ 物件導向：真正的「繼承」
class V9 {
  async requireUser() { /* ... */ }
}

class V10 extends V9 {
  // ← V9.requireUser() 改了，V10 自動跟著改（執行時綁定）
  async requireRole() { /* ... */ }
}

// ❌ Git：不是真正的「繼承」
feat/v9  → commit "fix: requireUser bug"
feat/v10 → 不會自動得到這個 fix
         → 必須手動 git merge（開發時手動操作）
         → 而且可能會 conflict
```

**關鍵差異**：

- **物件導向**：繼承是「執行時自動綁定」，父類改了子類自動生效
- **Git**：merge 是「開發時手動操作」，而且會有 conflict

#### 🌍 真實世界的版本管理策略

##### 產品開發：主動維護少數版本

大多數產品**不會**同時維護 v7/v8/v9/v10：

```bash
# 典型的產品開發流程
main (v10.2.1)  ← 當前唯一版本
  └── 持續開發、修 bug、加功能

v9.x (EOL - End of Life)  ← 不再維護
v8.x (EOL)
v7.x (EOL)

# 用戶必須升級到 v10，舊版本不再支援
```

##### 長期支援版本（LTS）

只有在「長期支援版本」策略下才會維護多版本：

```bash
main (v22.x)        ← 最新實驗版
v20.x (Active)      ← 當前穩定版
v18.x (LTS)         ← 長期支援，只修 critical bug
v16.x (EOL)         ← 不再支援
```

**Bug fix 如何同步？**

```bash
# 在 LTS 版本修正安全漏洞
cd v18
git commit -m "security: fix CVE-2024-1234"

# 同步到其他版本（手動！）
cd v20
git cherry-pick <commit-hash>  # ← 挑選特定 commit
# 解決 conflict（如果有）

cd main
git cherry-pick <commit-hash>
# 解決 conflict（如果有）

→ 不是自動繼承！是手動同步！
```

```
Node.js 版本線（2024）：

```

#### 🔄 DRY 原則與「重做輪子」

##### DRY 原則的真正含義

**DRY (Don't Repeat Yourself)** 是什麼？

```
❌ 常見誤解：「不要重複任何程式碼」
✅ 真正含義：「每一個知識點在系統中應該有唯一、明確的表達」
```

**出處**：《The Pragmatic Programmer》（務實的程式設計師）

> "Every piece of knowledge must have a single, unambiguous, authoritative representation within a system."

**重點**：

- 不是「程式碼」的重複
- 是「知識」的重複
- 強調「單一事實來源」(Single Source of Truth)

---

##### 什麼是「重做輪子」(Reinventing the Wheel)？

**重做輪子**是指：重新實作已經有成熟解決方案的問題。

```typescript
// ❌ 重做輪子：自己寫 HTTP client
async function myFetch(url: string) {
  // ... 手動實作 HTTP 請求
  // 問題：fetch API 已經存在且成熟
}

// ✅ 使用現有輪子
const res = await fetch(url);

// ❌ 重做輪子：自己寫日期處理
function formatDate(date: Date) {
  // ... 手動處理時區、格式
  // 問題：date-fns、dayjs 已經很成熟
}

// ✅ 使用現有輪子
import { format } from "date-fns";
const formatted = format(date, "yyyy-MM-dd");
```

**重做輪子的特徵**：

1. 社群已有成熟的解決方案（npm 套件、標準 API）
2. 你的實作不會比現有方案更好
3. 浪費時間在「已解決的問題」上
4. 通常會有更多 bug

---

##### Clone 策略是否違反 DRY 原則？

**問題**：v9 修了一個 bug，v10 也要重新修一次，這不是違反 DRY 嗎？

**答案**：看情況。讓我們分析幾種情境。

---

###### 情境 1：真正的「知識重複」（違反 DRY）❌

```typescript
// v9/util.ts
export function toTaipeiDateTime(isoString: string): string {
  const date = new Date(isoString);
  return date.toLocaleString("zh-TW", { timeZone: "Asia/Taipei" });
}

// v10/util.ts（複製貼上）
export function toTaipeiDateTime(isoString: string): string {
  const date = new Date(isoString);
  return date.toLocaleString("zh-TW", { timeZone: "Asia/Taipei" });
}

// 問題：這個「知識」（如何轉換台北時間）重複了
// 如果發現 bug（例如夏令時間處理錯誤），要改兩次
```

**這是違反 DRY？**

- ✅ **是**，如果這兩個版本在同一個「系統」中
- ❌ **不是**，如果它們是兩個獨立的「系統」

**關鍵**：V9 和 V10 是同一個系統嗎？

---

###### 情境 2：獨立的系統（不違反 DRY）✅

```typescript
// 專案 A（早餐店系統）
function toTaipeiDateTime(iso: string) {
  /* ... */
}

// 專案 B（圖書館系統）
function toTaipeiDateTime(iso: string) {
  /* ... */
}

// 這兩個是不同專案，重複的程式碼沒問題
```

**同理**：V9 和 V10 是兩個獨立的教學範例

```
V9 系統（bf_v9 schema）
  └── 一個完整、凍結的系統
  └── 目標：展示 Better Auth 整合

V10 系統（bf_v10 schema）
  └── 另一個完整、凍結的系統
  └── 目標：展示 RBAC 權限管理

它們不是「同一個系統的兩個部分」
而是「兩個獨立的完整範例」
```

**類比**：

```
Django 官方教學的 Tutorial 1（投票系統）
Django 官方教學的 Tutorial 2（部落格系統）
→ 兩個教學都會重複教「如何建立 model」
→ 這不是違反 DRY，因為它們是獨立的教學範例
```

---

###### 情境 3：架構層級的差異（不違反 DRY）✅

```typescript
// v9/backend.ts（無 RBAC）
async function requireUser(request: Request) {
  const user = await getCurrentUser(request);
  if (!user) throw unauthorized();
  return user;
}

// v10/backend.ts（有 RBAC）
async function requireUser(request: Request) {
  const user = await getCurrentUser(request);
  if (!user) throw unauthorized();
  return user;
}

// 看起來重複？但...

// v10 還有這些（v9 沒有）
function requireRole(user: SessionUser, role: Role) {
  /* ... */
}
function requireAnyRole(user: SessionUser, roles: Role[]) {
  /* ... */
}
```

**為什麼不違反 DRY？**

V10 的 `requireUser` 雖然和 V9 類似，但它在**不同的架構脈絡**中：

```
V9 架構：
  ├── requireUser() ← 終點（只驗證登入）
  └── 沒有角色概念

V10 架構：
  ├── requireUser() ← 起點（驗證登入）
  ├── requireRole() ← 第二層（驗證角色）
  └── requireAnyRole() ← 第三層（驗證多角色）
```

這不是「知識重複」，而是「架構演化」。

---

##### Clone 策略是否「重做輪子」？

**問題**：V9 修了 bug，V10 要重新修，這不是重做輪子嗎？

**答案**：不是。

**重做輪子的定義**：

- 重新實作「已有成熟解決方案」的問題
- 關鍵字：**解決方案已存在**（npm、標準庫、社群方案）

**Clone 策略的情況**：

```typescript
// V9 修正了一個 session 過期的 bug
// V10 面臨同樣的 bug

選項 1：Merge from v9（繼承）
  └── 問題：conflict、架構不相容

選項 2：手動複製修正（clone 策略）
  └── 不是「重做輪子」，而是「移植已知解決方案」

選項 3：真的「重做輪子」
  └── 忽略 v9 的修正，從頭研究問題、設計新解法
  └── 這才是真正的「重做輪子」
```

**區別**：

- ❌ 重做輪子：從零開始解決「已解決的問題」（浪費）
- ✅ 移植解決方案：將「已知的解法」應用到另一個系統（合理）

---

##### 真實案例：Node.js 的多版本維護

```
Node.js v18 發現 OpenSSL 漏洞
├── 修正並發布 v18.20.1
├── Cherry-pick 到 v20 → 發布 v20.15.1
└── Cherry-pick 到 v22 → 發布 v22.5.1

問題：這違反 DRY 原則嗎？
```

**分析**：

```typescript
// 三個版本都有類似的修正
// v18/deps/openssl/ssl_lib.c
if (ctx->session_timeout < 0) {  // ← bug fix
  return -1;
}

// v20/deps/openssl/ssl_lib.c（相同修正）
if (ctx->session_timeout < 0) {  // ← 複製過來的
  return -1;
}

// v22/deps/openssl/ssl_lib.c（相同修正）
if (ctx->session_timeout < 0) {  // ← 複製過來的
  return -1;
}
```

**這不違反 DRY，因為**：

1. 三個版本是**獨立維護的發行版**
2. 不是「同一個系統的不同部分」
3. 目標是讓每個版本都能**獨立運作**

---

##### 什麼時候「重複」是可接受的？

**✅ 可接受的重複**：

**1. 獨立系統之間**

```typescript
// 專案 A
function validateEmail(email: string) {
  /* ... */
}

// 專案 B（完全不同的專案）
function validateEmail(email: string) {
  /* ... */
}

// 沒問題：兩個獨立專案
```

**2. 不同抽象層級**

```typescript
// 底層工具函數
function formatDate(date: Date): string {
  /* ... */
}

// 業務邏輯層（包裝了底層）
function formatOrderDate(order: Order): string {
  return formatDate(order.createdAt); // 使用底層
}

// 雖然都在「格式化日期」，但層級不同
```

**3. 暫時的重複（準備重構）**

```typescript
// feature-A.ts
function parseConfig(json: string) {
  /* ... */
}

// feature-B.ts（先複製，之後抽出）
function parseConfig(json: string) {
  /* ... */
}

```

```typescript
// tutorial-01/（基礎教學）

  /* ... */
```

**❌ 不可接受的重複**：

**1. 同一系統內的知識重複**

```typescript
// ❌ 壞例子：同一個系統的兩個檔案
// users/validator.ts
function isValidEmail(email: string) {
  return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email);
}

// orders/validator.ts（複製貼上）
function isValidEmail(email: string) {
  return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email);
}

// 問題：「如何驗證 email」這個知識重複了
// 應該抽到 shared/validators.ts
```

**2. Magic Number / Magic String 重複**

```typescript
// ❌ 壞例子
if (user.role === "admin") {
  /* ... */
} // 在 10 個地方重複
if (user.role === "admin") {
  /* ... */
}
if (user.role === "admin") {
  /* ... */
}

// ✅ 好例子
const ROLE_ADMIN = "admin"; // 單一定義
if (user.role === ROLE_ADMIN) {
  /* ... */
}
```

##### 🎯 Clone 策略的判斷標準

**問題**：V9 修了 bug，V10 要重新修，這可接受嗎？

**判斷標準**：

| 條件                      | 可接受 | 理由                   |
| ------------------------- | ------ | ---------------------- |
| V9 和 V10 使用不同 schema | ✅     | 兩個獨立系統           |
| V9 和 V10 不會同時修改    | ✅     | 不是同一個 codebase    |
| V9 凍結後不再改動         | ✅     | 不需要持續同步         |
| Bug fix 可選擇性移植      | ✅     | 不是強制繼承           |
| 架構已經分歧              | ✅     | 直接 merge 會 conflict |

**如果**：

- V9 和 V10 在同一個 monorepo 中
- 共用相同的 `shared/` 目錄
- 預期要長期同步維護
  → 那就**違反 DRY**，應該用 Merge 策略

**但現況是**：

- V9 和 V10 是獨立的專案目錄
- V9 會凍結（作為教學參考）
- V10 是新的實驗版本
  → **不違反 DRY**，Clone 策略合理

##### 💡 實務建議：如何在 Clone 策略中減少「重複工」

**1. 建立「移植清單」**

```markdown
# v9-to-v10-patches.md

## Critical Bug Fixes（必須移植）

- [ ] CVE-2024-1234: Session 安全漏洞 (v9 commit: abc123)
- [ ] Bug: toTaipeiDateTime 時區錯誤 (v9 commit: def456)

## Nice-to-Have（可選移植）

- [ ] Perf: 優化 SQL 查詢 (v9 commit: ghi789)
- [ ] UX: 錯誤訊息改善 (v9 commit: jkl012)

## Skip（不移植）

- [ ] Feature: Email 通知 (v10 用 Google OAuth，不需要)
```

**2. 標記共用邏輯**

```typescript
// shared/utils.ts
/**
 * 轉換為台北時間
 *
 * @sync v7, v8, v9, v10  ← 標記：這段邏輯在所有版本共用
 * @lastUpdate 2024-05-18
 */
export function toTaipeiDateTime(iso: string): string {
  // ...
}
```

**3. 自動化測試確保一致性**

```typescript
// v9/util.test.ts

  expect(toTaipeiDateTime("2024-03-10T02:00:00Z")).toBe("2024-03-10 10:00:00");
```

##### ✨ 結論

**Q1: Clone 策略違反 DRY 原則嗎？**

**A**: **不違反**，因為：

- DRY 的真正含義是「每個知識在**同一個系統**中有唯一表達」
- V9 和 V10 是**兩個獨立系統**（不同 schema、凍結的教學範例）
- 「重複」發生在「系統之間」，不是「系統內部」

**Q2: V9 修 bug，V10 要重修，這是重做輪子嗎？**

**A**: **不是**，因為：

- 重做輪子 = 重新發明「已有成熟解決方案」的問題
- 移植 v9 的修正到 v10 = 應用「已知解法」到另一個系統
- 這是「移植」(porting)，不是「重做」(reinventing)

**Q3: 什麼時候應該擔心違反 DRY？**

**A**: 當重複發生在**同一個系統內部**時：

- ❌ 同一個專案的多個檔案重複相同邏輯
- ❌ Magic string 到處散落
- ❌ 複製貼上導致維護困難

**Q4: Clone 策略的代價值得嗎？**

**A**: 對教學場景來說，**值得**：

- ✅ 穩定性 > 維護效率
- ✅ 清晰度 > DRY 原則
- ✅ 獨立性 > 程式碼重用

**類比**：

```
教科書第 7 章和第 10 章都解釋了「迴圈」
→ 這不是違反 DRY，而是「教學必要的重複」
→ 每一章都應該是「獨立完整的學習單元」
```

**工程決策的本質**：

> 沒有絕對的「對」或「錯」，只有「適合」或「不適合」。  
> Clone 策略在教學場景下是最佳選擇，但在產品開發中可能是反模式。  
> 理解 Trade-off，根據目標選擇策略。

---

#### ✨ 最終答案

**Q: Git 版控系統真的可以做到繼承嗎？**

**A: 技術上可以，但實務上不建議**，因為：

1. ❌ **無法選擇性繼承**（全拿或全不拿）
2. ❌ **Merge conflict 會頻繁發生**（架構分歧越大，conflict 越多）
3. ❌ **資料庫 schema 會衝突**（不同版本改同一張表）
4. ❌ **依賴版本會衝突**（套件升級導致 breaking change）

**Q: 你們的 Clone 策略是錯的嗎？**

**A: 不，這是教學場景的最佳實踐！**

因為：

- ✅ 教學需要**穩定的參考版本**（凍結）
- ✅ 學生需要**清晰的學習階梯**（可對照）
- ✅ 實驗需要**安全的沙箱**（不怕弄壞）
- ✅ 版本間**架構差異大**（RBAC vs 無 RBAC）

**產品開發 vs 教學專案：不同的場景，不同的最佳策略**

| 策略  | 產品開發              | 教學專案      |
| ----- | --------------------- | ------------- |
| Merge | ✅ 主要策略           | ❌ 會造成混亂 |
| Clone | ⚠️ 少用（僅重大重構） | ✅ 最佳策略   |
| 原因  | 需要持續演化          | 需要穩定教材  |

## 技術方案選型

### 方案一：Better Auth 內建 RBAC

Better Auth 提供 `organization` plugin，支援角色權限管理。

#### 優點

✅ **快速整合** - 與現有 Better Auth 無縫整合  
✅ **自動遷移** - Drizzle adapter 自動處理 DB schema  
✅ **型別安全** - 完整的 TypeScript 型別定義  
✅ **前端支援** - 提供 React hooks  
✅ **標準化** - 遵循業界慣例（Organization → Role → Permission）

#### 缺點

❌ **過度設計** - 引入 `organization` 概念（單店場景不需要）  
❌ **學習曲線** - 需要理解 Better Auth 的權限模型  
❌ **彈性受限** - 角色定義必須符合 Better Auth 的資料結構  
❌ **DB 表增加** - 至少新增 3-4 張表（organization、member、role、permission）  
❌ **查詢複雜** - 權限檢查需要 JOIN 多張表  
❌ **文檔不足** - Better Auth RBAC 文檔較少，社群範例有限

#### 實作預覽

```typescript
// 需要建立 organization（即使只有一間店）
const organization = await auth.api.createOrganization({
  name: "聯大早餐店",
  slug: "nuu-breakfast",
});

// 為使用者分配角色
await auth.api.setRole({
  userId: user.id,
  organizationId: organization.id,
  role: "chef",
});

// 權限檢查
const hasPermission = await auth.api.hasPermission({
  userId: user.id,
  permission: "order:update",
});
```

### 方案二：自行管理 RBAC ✅ (採用)

在 `user` 表加 `roles` 欄位，業務層自行處理權限邏輯。

#### 優點

✅ **簡單直觀** - 直接在資料結構中管理角色  
✅ **完全控制** - 權限邏輯完全自主，容易理解與調整  
✅ **高效查詢** - 無需 JOIN，直接從 session 取 roles  
✅ **契合現況** - 符合三層架構（contracts → route-schemas → backend）  
✅ **易於測試** - 權限邏輯集中，測試覆蓋簡單  
✅ **教學友善** - 學生容易理解整個權限流程

#### 缺點

❌ **需自行實作** - 權限檢查邏輯需手寫  
❌ **缺少標準化** - 沒有業界標準模型可參考  
❌ **可能重複造輪** - 如果未來需要複雜權限，需大改

#### 選擇理由

1. **教學性質專案** → 簡單直觀的實作更有教育價值
2. **單店場景** → 不需要 organization 概念
3. **五個固定角色** → 不需要動態權限系統
4. **現有架構契合度高** → 延續 contracts-first 設計
5. **效能更好** → roles 存在 session 中，無需每次查 DB

### Q1: 預設角色與升級機制

**問題**：新註冊使用者預設為 `customer`，是否需要審核機制才能升級為 `staff` / `chef`?

**決策**：✅ 需要審核機制

**設計方案**：

```typescript
  id: number;
  userId: string;

  reason: string;
```

### 3. Guards 層（權限檢查）

```typescript
// shared/guards.ts

import type { SessionUser, Role } from "./contracts.ts";

/**
 * 檢查使用者是否擁有指定角色
 */
export function hasRole(user: SessionUser, role: Role): boolean {
  return user.roles.includes(role);
}

/**
 * 檢查使用者是否擁有任一指定角色（OR 邏輯）
 */
export function hasAnyRole(user: SessionUser, roles: Role[]): boolean {
  return roles.some((role) => user.roles.includes(role));
}

/**
 * 檢查使用者是否擁有所有指定角色（AND 邏輯）
 */
export function hasAllRoles(user: SessionUser, roles: Role[]): boolean {
  return roles.every((role) => user.roles.includes(role));
}

/**
 * 要求使用者擁有指定角色，否則拋出 403 錯誤
 */
export function requireRole(user: SessionUser, role: Role): void {
  if (!hasRole(user, role)) {
    throw new Response(
      JSON.stringify({
        error: "Forbidden",
        message: `Required role: ${role}`,
      }),
      {
        status: 403,
        headers: { "Content-Type": "application/json" },
      },
    );
  }
}

/**
 * 要求使用者擁有任一指定角色，否則拋出 403 錯誤
 */
export function requireAnyRole(user: SessionUser, roles: Role[]): void {
  if (!hasAnyRole(user, roles)) {
    throw new Response(
      JSON.stringify({
        error: "Forbidden",
        message: `Required one of roles: ${roles.join(", ")}`,
      }),
      {
        status: 403,
        headers: { "Content-Type": "application/json" },
      },
    );
  }
}

/**
 * 檢查使用者是否為資源擁有者
 */
export function isResourceOwner(
  user: SessionUser,
  resourceUserId: string,
): boolean {
  return user.id === resourceUserId;
}

/**
 * 檢查是否為資源擁有者或擁有管理角色
 */
export function canAccessResource(
  user: SessionUser,
  resourceUserId: string,
  adminRoles: Role[] = ["admin", "owner"],
): boolean {
  return isResourceOwner(user, resourceUserId) || hasAnyRole(user, adminRoles);
}
```

```typescript
    const updatedOrder = await store.updateOrder(orderId, body);

    if (!updatedOrder) {
      set.status = 404;
      return { error: "Order not found" };
    }

    return { data: updatedOrder };
  },
  {
    params: updateOrderParamsSchema,
    body: updateOrderBodySchema,
    detail: {
      tags: ["orders"],
      summary: "Update order (chef/owner/admin only)",
    },
    response: { 200: orderResponseEnvelopeSchema, 403: apiErrorResponseSchema },
  },
```

## 前端權限控制

### 1. Auth Context 擴充

```typescript
// frontend/src/contexts/AuthContext.tsx

import { createContext, useContext, useEffect, useState } from "react";
import type { SessionUser, Role } from "../../../shared/contracts";

interface AuthContextValue {
  user: SessionUser | null;
  loading: boolean;
  error: string | null;
  hasRole: (role: Role) => boolean;
  hasAnyRole: (roles: Role[]) => boolean;
  hasAllRoles: (roles: Role[]) => boolean;
  signOut: () => Promise<void>;
}

const AuthContext = createContext<AuthContextValue | undefined>(undefined);

export function AuthProvider({ children }: { children: React.ReactNode }) {
  const [user, setUser] = useState<SessionUser | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<string | null>(null);

  // 載入使用者資訊
  useEffect(() => {
    async function loadUser() {
      try {
        const res = await fetch("/api/auth/session", {
          credentials: "include",
        });

        if (res.ok) {
          const data = await res.json();
          setUser(data.user);
        } else {
          setUser(null);
        }
      } catch (err) {
        setError("Failed to load user");
        setUser(null);
      } finally {
        setLoading(false);
      }
    }

    loadUser();
  }, []);

  // 權限檢查函數
  const hasRole = (role: Role): boolean => {
    return user?.roles.includes(role) ?? false;
  };

  const hasAnyRole = (roles: Role[]): boolean => {
    return roles.some(role => user?.roles.includes(role)) ?? false;
  };

  const hasAllRoles = (roles: Role[]): boolean => {
    return roles.every(role => user?.roles.includes(role)) ?? false;
  };

  const signOut = async () => {
    await fetch("/api/sign-out", {
      method: "POST",
      credentials: "include",
    });
    setUser(null);
  };

  return (
    <AuthContext.Provider
      value={{ user, loading, error, hasRole, hasAnyRole, hasAllRoles, signOut }}
    >
      {children}
    </AuthContext.Provider>
  );
}

export function useAuth() {
  const context = useContext(AuthContext);
  if (!context) {
    throw new Error("useAuth must be used within AuthProvider");
  }
  return context;
}
```

```typescript
// frontend/src/App.tsx

import { BrowserRouter, Routes, Route, Navigate } from "react-router-dom";
import { useAuth } from "./contexts/AuthContext";

function ProtectedRoute({
  children,
  roles
}: {
  children: React.ReactNode;
  roles?: Role[]
}) {
  const { user, loading, hasAnyRole } = useAuth();

  if (loading) return <div>Loading...</div>;
  if (!user) return <Navigate to="/login" replace />;
  if (roles && !hasAnyRole(roles)) {
    return <Navigate to="/forbidden" replace />;
  }

  return <>{children}</>;
}

function App() {
  return (
    <AuthProvider>
      <BrowserRouter>
        <Routes>
          <Route path="/login" element={<LoginPage />} />

          {/* 一般使用者路由 */}
          <Route path="/menu" element={<MenuPage />} />
          <Route path="/my-orders" element={
            <ProtectedRoute>
              <MyOrdersPage />
            </ProtectedRoute>
          } />

          {/* 店員以上路由 */}
          <Route path="/orders" element={
            <ProtectedRoute roles={["staff", "chef", "owner", "admin"]}>
              <AllOrdersPage />
            </ProtectedRoute>
          } />

          {/* 管理員路由 */}
          <Route path="/admin/*" element={
            <ProtectedRoute roles={["admin"]}>
              <AdminPanel />
            </ProtectedRoute>
          } />
        </Routes>
      </BrowserRouter>
    </AuthProvider>
  );
}
```

## 安全性考量

### 後端安全

1. **權限檢查必須在後端執行**
   - 前端權限控制只是 UX 優化
   - 真正的安全邊界在後端 API

2. **Session 驗證**
   - 每個需要權限的 API 都必須先呼叫 `requireUser()`
   - 不可信任前端傳來的角色資訊

3. **資源擁有者檢查**
   - 除了角色檢查，還要確認資源歸屬
   - 例如：顧客只能查看「自己的」訂單

4. **角色升級審核**
   - 重要角色（`owner`、`admin`）只能由 admin 直接分配
   - 申請升級需要審核機制

---

### 前端安全

1. **不可繞過的後端檢查**
   - 即使前端隱藏了按鈕，API 仍須驗證權限
   - 攻擊者可直接呼叫 API

2. **敏感資訊不顯示**
   - 根據角色決定 API response 內容
   - 例如：顧客查訂單時不回傳其他人的資料

3. **HTTPS 強制使用**
   - Session cookie 必須設定 `Secure` 標記
   - 防止中間人攻擊

## 常見問題 FAQ

### Q1: 為什麼不使用 Better Auth 的 RBAC?

**A**: 因為：

1. 本專案是教學性質，自行實作更有教育價值
2. Better Auth RBAC 引入 `organization` 概念，對單店場景過度設計
3. 自行管理更靈活，容易根據業務需求調整
4. 效能更好（無需 JOIN 多張表）

---

### Q2: 為什麼使用 array 而非 many-to-many?

**A**: 因為：

1. 角色數量固定且少（5 個）
2. PostgreSQL 原生支持 array 操作
3. 查詢效率高（無需 JOIN）
4. 實作簡單，符合教學需求

如果未來角色數量增加或需要更複雜的權限管理，可重構為 many-to-many。

---

### Q3: 前端權限控制是否安全?

**A**: ⚠️ **前端權限控制不是安全邊界**！

- 前端控制只是為了提升 UX，避免使用者看到無權操作的按鈕
- 真正的權限檢查必須在後端 API 執行
- 攻擊者可以繞過前端直接呼叫 API

---

### Q4: 如何建立第一個 admin 使用者?

**A**: 有三種方式：

**方式 1：手動 SQL**

```sql
UPDATE bf_v10.user
SET roles = ARRAY['admin', 'customer']
WHERE email = 'your-email@example.com';
```

**方式 2：執行 script**

```bash
bun run scripts/create-admin.ts your-email@example.com
```

**方式 3：環境變數**

```env
INITIAL_ADMIN_EMAIL=your-email@example.com
```

在 server 啟動時自動檢查並升級。

---

### Q5: 多角色使用者的權限如何計算?

**A**: 採用 **OR 邏輯**（聯集）

```typescript
// 使用者擁有 ["owner", "chef"]
hasRole("owner"); // ✅ true
hasRole("chef"); // ✅ true
hasRole("staff"); // ❌ false

hasAnyRole(["staff", "owner"]); // ✅ true（擁有其中一個）
hasAllRoles(["owner", "chef"]); // ✅ true（同時擁有兩個）
```

---

### Q6: 角色可以被移除嗎?

**A**: 可以，由 admin 透過 API 操作：

```typescript
PATCH /api/admin/users/:userId/roles
{
  "roles": ["customer"]  // 移除其他角色，只保留 customer
}
```

⚠️ **注意**：至少要保留一個角色（`customer` 是預設角色）。
