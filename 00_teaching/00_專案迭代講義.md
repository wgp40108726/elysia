# 早餐店前後端分離專案迭代講義

閱讀入口建議先參考 [README.md](/root/00_nsPrj/01_backEnd/06_elysia/00_demo01/00_teaching/README.md:1)，再依主線順序閱讀各份講義。

## 1. 講義定位

本講義整理此專案從初版到目前版本的演進脈絡，重點不只是「做了哪些功能」，而是說明：

- 為什麼當時要改版
- 每一版解決了什麼問題
- 架構是如何一步一步從示範型專案，走向較完整的前後端分離系統

補充說明：本專案資料夾目前不是 Git repository，因此本講義的版本脈絡，是根據專案內保留的版本檔案推回來整理，例如 `backend_01.ts`、`backend_02.ts`、`frontend/src/App_01.tsx`、`store/json/JsonFileStore_01.ts` 等。

---

## 2. 專案總體架構

此專案採用「開發分離、部署整合」模式。

### 開發階段

- 前端位於 `frontend/`，使用 `React + Vite`
- 後端位於專案根目錄，使用 `Elysia`
- 前端開發伺服器運行在 `http://localhost:5173`
- 後端 API 運行在 `http://localhost:3000`
- Vite 透過 proxy 將 `/api` 轉發到後端

### 部署階段

- 前端 build 後輸出到 `public/`
- 後端透過 `@elysiajs/static` 直接提供靜態資產
- 因此部署時只要啟動後端，就能同時提供 API 與 Web App

### 這種架構為什麼適合教學

- 可以清楚區分前端與後端責任
- 可以讓理解 API 契約的重要性
- 可以同時示範「分離開發」與「整合部署」
- 比單體頁面更接近真實產品開發模式

## 3. 版本演進總覽

目前已完成的演進可以先看成七個主要版本：

1. `V1 初版原型`：先把點餐流程跑起來
2. `V2 資料持久化版`：讓資料不再只存在記憶體
3. `V3 菜單內容升級版`：讓資料模型支援圖片與描述
4. `V4 體驗與部署修正版`：處理前端狀態延續與靜態資產問題
5. `V5 使用者登入與個人訂單版`：加入使用者登入、個人訂單與歷史紀錄
6. `V6 OpenAPI ／ Swagger 文件版`：把既有 route schema 輸出成可瀏覽的 API 文件
7. `V7 目前版本／正式入口收斂版`：把 V6 收斂成專案目前實際執行的主入口

若把接下來已經明確的規劃方向也一起納入，則後續還會接上兩個版本：

8. `V8 Drizzle + Neon 資料庫升級版`：把 JSON store 換成正式資料庫
9. `V9 Better Auth + Google provider 版`：把示範型登入升級成正式 auth 架構

- 目前打開的是 `backend_03.ts`

### 一個實用判斷原則

若正在讀某個版本，但看到下面這種情況，就要先停一下確認：

- 檔名是 `_02`，但型別卻 import 到未加後綴的 `contracts.ts`
- 檔名是 `_03`，但 store 卻 import 到未加後綴的 `Store.ts`
- 講義說是 V3，但 source 實際連到的是 V5 之後共用的資料模型

這通常不是看錯業務邏輯，而是因為「版本快照檔」和「目前正式入口檔」混在一起了。

處理方式很簡單：

- 先回來查這張對照表
- 確認該版應該追哪一組檔案
- 再回去讀 source

這樣比較不會在不同版本的資料模型之間來回跳。

## 5. 各版本迭代說明

### V1 初版原型

### 對應檔案

- `backend_01.ts`
- `shared/contracts_01.ts`
- `frontend/src/App_01.tsx`

### 版本目標

先建立一個最小可運作的早餐店點餐系統，證明前後端分離架構可行。

### 當時已完成的能力

- 後端提供菜單 API
- 後端提供建立訂單、查詢訂單、更新訂單品項 API
- 前端可以載入菜單
- 使用者可以加入購物車
- 可以清空購物車
- 可以看到目前購物車總額

### 當時的資料特性

- 菜單與訂單都存在記憶體
- 伺服器重啟後資料會消失
- 菜單欄位只有 `name`、`price`、`category`
- 訂單不綁定使用者

### 為什麼先做這一版

因為教學與原型開發最重要的是先把核心流程跑通。若一開始就加入帳號、資料庫、圖片、權限，會讓系統過重，也難以抓到主線。

### 這一版解決了什麼問題

- 確認前後端 API 溝通流程成立
- 建立共享型別契約的基礎
- 讓點餐流程有一個可以展示的最小成果

### 這一版的限制

- 資料不持久
- 沒有使用者概念
- 沒有訂單歷史
- 菜單資料過於簡化，不利於前端做完整商品卡片

### 教學重點

- MVP 思維
- API 設計入門
- 前後端分離的第一步不是複雜，而是先把資料流打通

---

### V2 資料持久化版

### 對應檔案

- `backend_02.ts`
- `store/Store_01.ts`
- `store/json/JsonFileStore_01.ts`

### 版本目標

把原本直接寫在 `backend_01.ts` 裡的資料與商業邏輯抽離，改成可持久化的 Store 架構。

### 主要變更

- 新增 `Store` 介面
- 新增 `JsonFileStore`
- 菜單與訂單資料改存入 `data/store.json`
- 後端路由不再直接操作陣列，而是呼叫 store 方法

### 為什麼要做這次迭代

因為 V1 雖然能展示流程，但不適合繼續擴充：

- 伺服器一重啟，資料全部消失
- 路由層直接管理資料，責任混在一起
- 未來若要換成資料庫，會很難替換

### 這一版解決了什麼問題

- 讓資料在重啟後仍可保留
- 將 API 層與資料層分離
- 建立抽象層，未來可以替換成資料庫版 store

### 教學意義

這一版很適合拿來講「架構重構不是為了炫技，而是為了降低未來修改成本」。

可清楚看到：

- `backend_01.ts` 是把所有事情寫在一起
- `backend_02.ts` 之後，程式開始有分層觀念

### 核心觀念

- Controller 與 Data Access 分離
- 介面導向設計
- 持久化是從 demo 走向可用系統的第一步

### V3 菜單內容升級版

### 對應檔案

- `backend_03.ts`
- `shared/contracts_02.ts`
- `store/Store_02.ts`
- `store/json/JsonFileStore_02.ts`

### 版本目標

讓菜單資料從「只有價格與分類」升級成「可以支援商品展示」的完整內容模型。

### 主要變更

- `MenuItem` 新增 `description`
- `MenuItem` 新增 `image_url`
- 建立 `normalizeMenuItem()`，兼容舊資料
- 新增較完整的預設菜單內容

### 為什麼要做這次迭代

因為 V2 的資料結構雖然可持久化，但仍然偏向後端資料示範，不足以支撐真正的前端商品展示介面。

如果前端要做卡片式 UI，至少要有：

- 商品圖片
- 商品描述
- 更完整的內容欄位

### 這一版解決了什麼問題

- 前端不再只能顯示文字清單
- 菜單資料模型更接近真實電商或點餐系統
- 舊資料可透過 normalize 機制平滑升級，避免格式不一致造成系統中斷

### 教學重點

- 資料模型會隨著需求成熟而擴充
- 欄位新增不只是「加兩個屬性」，還要思考舊資料相容性
- 真實系統的改版，通常會牽動 API、型別、資料檔與前端畫面

### V4 體驗與部署修正版

### 對應檔案

- `backend_04.ts`
- `frontend/src/App_02.tsx`
- `store/json/JsonFileStore_03.ts`
- `store/json/JsonFileStore_04.ts`
- `store/json/JsonFileStore_05.ts`

### 版本目標

改善使用者操作體驗，並修正前端 build 後的靜態資產與 SPA 路由問題。

### 主要變更

- 前端將 `orderId` 存入 `localStorage`
- 重新整理頁面後可以恢復尚未完成的訂單
- 預設菜單從遠端圖片改為專案本地圖片
- 圖片路徑逐步從中文檔名調整為英文檔名
- SPA fallback 改為先判斷靜態檔是否存在，再決定是否回傳 `index.html`

### 為什麼要做這次迭代

V3 雖然能顯示圖片與描述，但實際使用時會遇到兩類問題：

第一類是使用者體驗問題：

- 重新整理後購物車狀態消失
- 使用者會覺得系統不穩定

第二類是部署問題：

- 遠端圖片連線不穩定
- 靜態資產在部署環境可能找不到
- SPA 路由與靜態檔回應邏輯容易互相衝突

### 這一版解決了什麼問題

- 提升前端狀態延續性
- 降低外部圖片來源依賴
- 讓 build 後的前端資產更穩定
- 讓前後端整合部署時的靜態資源行為更正確

### 教學重點

- 功能做完不代表產品可用，還要考慮操作連續性
- 部署問題通常會逼出下一波架構修正
- 真實專案的迭代，常常不是增加功能，而是修正「看不見但很痛」的問題

### V5 使用者登入與個人訂單版

### 對應檔案

- `backend_05.ts`
- `shared/contracts.ts`
- `store/Store.ts`
- `store/json/JsonFileStore.ts`
- `frontend/src/App.tsx`

### 版本目標

把系統從「單純的購物車 demo」提升為「有使用者脈絡的早餐點餐系統」。

### 主要變更

- 新增登入 API：`/api/auth/login`
- 新增 `User` 型別與預設使用者資料
- 訂單新增 `userId`
- 新增目前進行中訂單查詢：`/api/orders/current`
- 新增歷史訂單查詢：`/api/orders/history`
- 新增送出訂單 API：`/api/orders/:id/submit`
- 訂單新增狀態區分：`pending`、`submitted`
- 前端加入登入畫面
- 前端可查看自己的歷史訂單
- 前端購物車操作必須綁定登入使用者

### 為什麼要做這次迭代

因為前面的版本仍然把「訂單」視為系統層級資料，而不是「某位使用者的訂單」。這樣雖然方便示範，但不符合真實業務情境。

真實系統至少要回答以下問題：

- 這張訂單是誰的
- 使用者重新登入後，能不能找回自己的未完成訂單
- 已送出的訂單，能不能查詢歷史紀錄
- 不同使用者之間，能不能避免彼此誤改資料

### 這一版解決了什麼問題

- 訂單開始具有歸屬性
- 前端可依登入者載入對應資料
- 已送出與未送出訂單可分流管理
- API 開始有基本權限與狀態檢查觀念

### 這一版代表的架構成熟度

到了 V5，專案已不再只是「前後端串接展示」，而是具備以下特徵：

- 有共享契約
- 有資料儲存抽象層
- 有前端獨立開發流程
- 有整合部署流程
- 有使用者資料模型
- 有訂單生命週期概念

這代表專案已經從教學型 demo，邁向小型產品雛形。

### 教學重點

- 使用者身分會改變整個資料模型設計
- 訂單不只是資料集合，而是有生命週期的業務物件
- 權限、狀態、歷史查詢，是系統成熟的重要分水嶺

### 這一版仍然是示範型登入

雖然 V5 已經加入登入畫面與使用者概念，但它本質上仍是「示範型登入」，還不是正式產品等級的驗證架構。

目前實作特徵如下：

- 後端使用 `/api/auth/login` 做帳密驗證
- 前端把登入後的使用者資料存進 `localStorage`
- 後續 API 透過 `query.userId` 或 `body.userId` 判斷資料歸屬
- 後端沒有真正驗證「這次請求是否真的由該使用者發出」

這種做法適合教學 demo，因為流程簡單、容易理解；但若要正式上線，會有幾個明顯限制：

- 前端可自行修改 `localStorage` 內容
- API 若只相信前端傳來的 `userId`，安全性不足
- `User` 型別仍保留 `password` 欄位，表示帳號系統仍是自行管理
- 缺少真正的 session 驗證、cookie 管理、OAuth callback 與帳號綁定機制

### V6 OpenAPI ／ Swagger 文件版

### 對應檔案

- `backend_06.ts`
- `shared/contracts.ts`
- `store/Store.ts`
- `store/json/JsonFileStore.ts`
- `package.json`
- `test_03.rest`

### 版本目標

把 V5 已經存在的 route schema 進一步文件化，讓 API contract 不只存在程式裡，也能直接被瀏覽與測試。

### 主要變更

- 導入 `@elysiajs/openapi`
- 新增 `/openapi` 與 `/openapi/json`
- 為主要 route 補上 `detail.tags`
- 為主要 route 補上 `summary` 與 `description`
- 將 `OPTIONS` 與 SPA fallback 等非教學主線路由從文件中隱藏
- 讓 `test_03.rest` 可以直接驗證 OpenAPI 相關端點

### 為什麼要做這次迭代

到了 V5，系統其實已經有不少 `params / query / body / response` schema，但這些資訊主要還是散落在 route 定義中。協作者與開發者如果只看 API 使用方式，仍然要直接翻 source code。

因此 V6 要解決的，不再是「有沒有 schema」，而是「這些 schema 能不能被看見、被瀏覽、被當成教學與協作的共同契約」。

### 這一版解決了什麼問題

- API contract 從程式內部知識，變成可直接打開的文件
- route 的用途、分類與輸入輸出更加明確
- 測試、前端串接與教學示範都有更直接的依據
- 後續若再改 API，比較容易看出文件是否也同步變動

### 和 V5 最大差異

V5 比較像：

```text
共享型別 + route schema + 開發者自己讀 source
```

V6 則進一步變成：

```text
共享型別 + route schema + 可輸出的 OpenAPI 文件
```

### 教學重點

- schema 的價值不只是在驗證，也在於可溝通性
- API contract 要能被人閱讀，才容易成為團隊共識
- 文件若能從 route schema 直接輸出，維護成本會比手寫文件低很多

補充閱讀：

### V7 目前版本／正式入口收斂版

### 對應檔案

- `backend.ts`
- `shared/contracts.ts`
- `store/Store.ts`
- `store/json/JsonFileStore.ts`
- `README.md`
- `package.json`
- `test_03.rest`

### 版本目標

把 V6 已完成的 API 文件化成果收斂成專案目前真正執行的主入口，讓開發、建置、文件與測試都對準同一份 backend。

### 主要變更

- 將 `backend.ts` 定位為正式主入口
- `package.json` 的 `module`、`dev`、`build`、`start` scripts 全部對準 `backend.ts`
- `README.md` 也以 `backend.ts` 作為後端說明主體
- OpenAPI 端點成為目前版本的預設能力，而不是只存在於快照檔

### 為什麼要做這次迭代

有了 `backend_05.ts` 與 `backend_06.ts` 這種版本快照後，下一步就不是再發明一套新功能，而是把已經驗證可行的版本收斂成真正的專案主線。

這也是為什麼 `backend.ts` 與 `backend_06.ts` 在功能上幾乎一致，但角色不同：

- `backend_06.ts` 是版本快照
- `backend.ts` 是目前實際開發、執行、建置與部署都會用到的正式入口

### 這一版解決了什麼問題

- 降低「到底該跑哪個 backend 檔」的混淆
- 讓 README、scripts、測試與實際執行版本一致
- 讓 OpenAPI 文件成為目前主線的一部分，而不是旁支實驗
- 把版本快照整理回可維護的正式入口

### 目前版本仍然是示範型登入

雖然到 V7 為止，系統已經具備登入、個人訂單、歷史紀錄與 OpenAPI 文件，但它本質上仍是「示範型登入」，還不是正式產品等級的驗證架構。

目前實作特徵如下：

- 後端使用 `/api/auth/login` 做帳密驗證
- 前端把登入後的使用者資料存進 `localStorage`
- 後續 API 透過 `query.userId` 或 `body.userId` 判斷資料歸屬
- 後端沒有真正驗證「這次請求是否真的由該使用者發出」

這種做法適合教學 demo，因為流程簡單、容易理解；但若要正式上線，仍有幾個明顯限制：

- 前端可自行修改 `localStorage` 內容
- API 若只相信前端傳來的 `userId`，安全性不足
- `User` 型別仍保留 `password` 欄位，表示帳號系統仍是自行管理
- 缺少真正的 session 驗證、cookie 管理、OAuth callback 與帳號綁定機制
- 若 `localStorage` 留有舊登入資料（例如切換資料源或資料重置後），`/api/orders` 可能回 `401/403/404`，並表現為「加入購物車失敗」；前端應清理登入狀態並要求重新登入

### V7 在 Render 上的一個典型誤判案例

這一版在 Render 上很容易出現一種表面看似矛盾的現象：

- 按下「加入購物車」後，畫面跳出錯誤訊息
- 但重新登入後，剛才加入的品項其實已經存在購物車裡

這代表問題不一定出在資料沒有寫入，而可能出在「前端把異常路徑誤判成整次操作失敗」。

課堂上可用下面這組判讀順序帶著排查：

1. 先看重新登入後資料是否存在
2. 若存在，表示後端 mutation 與 JSON 寫入多半已成功
3. 再回頭檢查前端是否在 response 解析、登入狀態同步或例外處理時提早進入 `catch`
4. 最後再區分是「真的寫入失敗」還是「UI 沒有成功同步 server 最新狀態」

這個案例很適合拿來說明：

- 錯誤訊息不一定等於真正失敗點
- 前端狀態與後端狀態可能短暫不同步
- 在示範型登入架構中，`localStorage` 殘留資料會放大這種不同步問題

### V7 結尾就應先教第一次部署

教學節奏上，我會建議不要等到 V8 才第一次碰部署，而是就在 V7 結尾先做一次完整的 Render 部署。

原因很單純：

- V7 沒有資料庫，環境變數最少
- 學生可以先熟悉 Render 的基本流程
- 可以先建立 `Build Command`、`Start Command`、健康檢查與 API 驗證這些基本習慣

也就是說，V7 最適合拿來教：

1. 如何把 GitHub repo 接到 Render
2. 如何在 Render 的 `Node` 環境中使用 `bun` 指令
3. 如何填 `bun install && bun run build`
4. 如何用 `bun run start` 啟動
5. 如何驗證 `/health` 與 `/api/menu`

這樣做完之後，學生腦中會先有一個穩定基準：

```text
Render 基本流程已經會了
```

後面到 V8，就不用再重講整套平台操作。

### 後續建議的 V8：Drizzle + Neon 資料庫升級版

當 V7 已把主入口與 API 文件主線穩住後，下一步才是資料層升級。這一版的核心思路是：

- 把 `data/store.json` 換成正式資料庫
- 把 `JsonFileStore` 逐步換成 ORM 與 repository / query layer
- 讓 `menu`、`orders`、`order_items` 等資料進入 PostgreSQL
- 建立 migration 流程，讓資料模型升級不再靠手動改 JSON

這樣做的原因是：

- API 邊界已先穩住，資料庫重構時更容易分辨是資料層問題還是 API shape 問題
- 可以比較放心地替換底層資料來源，而不會同時把 API 契約也一起打亂

### 後續建議的 V9：Better Auth + Google Provider

當 V8 完成後，下一步再進入 auth 升級就會更合理。屆時的核心思路才會是：

- 把「身分驗證」交給 `Better Auth`
- 把 `Google` 當成登入來源之一
- 把「誰是目前登入者」改成由 server session 判定

也就是說，auth 升級不是取消，而是往後順延，建立在：

- API contract 已穩定
- 正式資料庫已到位

這兩個前提上。

### 適合本專案的後續實作順序

若從目前 V7 繼續往下走，最建議的順序是：

1. 先維持 `backend.ts` 與 OpenAPI 文件同步
2. 再導入 `Drizzle + Neon`
3. 再導入 `Better Auth + Google provider`
4. 最後把訂單 API 完整改成從 session 判斷目前使用者

這樣更符合「先穩 API 邊界、再換資料層、最後升級認證」的架構演進，也比較能讓看清楚每一步在解決什麼問題。

### 版本完成後就補部署差異，會比集中到最後更合理

若把部署教學重新整理成更穩定的節奏，我會建議這樣排：

1. `V7` 結尾：做第一次完整部署
2. `V8` 結尾：補講資料庫版的部署差異
3. `V9` 結尾：再補講 auth 版的部署差異
4. 之後每一版都維持同一節奏

這樣安排的好處是：

- 學生會把部署理解成「每版都要檢查的工程習慣」
- 不會把部署誤解成最後一次才做的獨立章節
- 每次只需要記住和上一版相比多了哪些調整

從 V7 到 V8，最典型的部署差異其實只有幾項：

1. `Build Command` 從 `bun install && bun run build`
   改成 `bun install && bun run db:migrate && bun run build`
2. 新增 `STORE_DRIVER=postgres`，讓 runtime 真正切到 PostgreSQL store
3. 新增 `DATABASE_URL`，並建議同步設定 `DATABASE_URL_MIGRATION`
4. 驗證時要從「只有 Web/API 可用」進一步變成「資料庫 migration 也已套用」

因此在主線上，V8 後、V9 前，比較好的里程碑應該是：

1. `feat/v8-drizzle-neon` 開發完成
2. 建立 PR，完成 code review 與 PM/PO 確認
3. 先在既有 Render 服務補上 V8 的部署調整
4. 再合併回 `main`
5. 由 `main` 觸發部署
6. 驗證健康檢查、核心 API 與資料庫連動
7. 部署確認後再打 `v8.0.0`
8. 再進入 V9（Better Auth + Google provider）

### 進入 V8 前，應先替 V7 留下 branch 與 tag

若流程是最理想狀態，確實應該是：

1. 先在 `feature/v7` 完成 V7
2. merge 回 `main`
3. 由 `main` 觸發部署
4. 再從 V7 基準往 V8 前進

但若實務上已經先把 V7 直接整理在 `main` 上，也不代表來不及補救。這時最合理的做法不是重寫 Git 歷史，而是立刻替目前這個 `main` 留下：

- 一條 `V7` 保留 branch
- 一個 `V7` 版本 tag

原因是：

- 等 V8 之後 merge 回 `main`，`main` 就不再代表 V7
- 若沒有 branch 或 tag，之後學生要回看 V7 會不直觀
- branch 適合之後補教材或小修
- tag 適合明確標記「這就是 V7 定版點」

若目前 `main` 就是 V7，可直接這樣做：

```bash
git switch main
git pull origin main
git branch v7-baseline
git tag -a v7.0.0 -m "V7 baseline before V8"
git push origin v7-baseline
git push origin v7.0.0
```

這樣之後的角色分工就很清楚：

- `main`：繼續往 V8、V9 前進
- `v7-baseline`：保留 V7 教學基準
- `v7.0.0`：標記 V7 定版點

建議之後每次都固定問三個問題：

1. `Build Command` 有沒有變
2. `Start Command` 有沒有變
3. 環境變數 / 外部服務有沒有新增

### `main` 改教材後，`V8 branch` 與學生本機不會自動更新

這也是很值得在課堂上順手教的一個實務觀念：

- `main` 有新 commit，不代表 `feat/v8-drizzle-neon` 會自動拿到
- 遠端 repo 有新 commit，不代表學生本機 clone 下來的資料夾會自動更新

也就是說，Git 預設永遠是：

```text
要自己抓更新
要自己同步分支
```

若學生只是想更新本機的 `main`，最基本做法是：

```bash
git switch main
git pull origin main
```

若學生正在 `feat/v8-drizzle-neon` 開發，想把最新教材同步進自己的 V8 分支，則可用：

```bash
git switch feat/v8-drizzle-neon
git fetch origin
git merge origin/main
```

若團隊已經熟悉較乾淨的歷史整理方式，也可改用：

```bash
git switch feat/v8-drizzle-neon
git fetch origin
git rebase origin/main
```

教學上，初期先教 `merge origin/main` 就夠，因為概念最直觀。

若要一步一步實作，請搭配附件：

- `02_3_V7_Render_首次部署教案手冊.md`
- `03_2_V8_合併主線與_Render_最小部署_CI_CD_教案手冊.md`
- `03_3_GitHub_PR_模板與審查清單.md`
- `03_4_V8_合併與部署_課堂評分Rubric.md`

## 5. 從版本演進看出來的設計思路

如果把這個專案的演進抽象化，可以看出一條很典型的系統成長路線：

### 第一階段：先跑通

先完成最小可展示流程，不急著追求完整架構。

### 第二階段：再拆層

當流程成立後，開始把資料層抽離，避免後面每改一次功能就牽動全部程式。

### 第三階段：補資料模型

當前端需求變多，原本簡化的資料結構就不夠用了，因此要升級欄位設計。

### 第四階段：修體驗與部署

產品一旦進入真實使用場景，就會暴露出重新整理、資產路徑、部署細節等問題。

### 第五階段：進入業務模型

最後才補上使用者、歷史訂單、狀態流轉，讓系統真正具備「業務意義」。

### 第六階段：把 API 契約文件化

在 V5 已有 route schema 的基礎上，再把它輸出成 OpenAPI / Swagger 文件，讓 API contract 變成可直接閱讀的共同介面。

### 第七階段：收斂成正式主入口

當文件化版本驗證完成後，再把 `backend.ts` 收斂為正式入口，讓 README、scripts、測試與實際執行都對準同一條主線。

### 第八階段：再升級資料基礎設施

當 API 邊界穩住後，再把 JSON store 換成 ORM 與正式資料庫，重構風險會低很多。

### 第九階段：最後升級驗證與權限

等資料庫與 API contract 都已穩定後，再把示範型登入升級成真正的 session 與第三方認證架構。

## 6. 適合上課時強調的結論

### 結論一

前後端分離不是目的，而是讓系統更容易分工、測試與部署。

### 結論二

版本迭代不應只看「多了哪些功能」，更要看「當時遇到什麼限制」。

### 結論三

好的迭代通常遵守這個順序：

- 先驗證可行
- 再整理架構
- 再擴充資料模型
- 再修正體驗與部署
- 再補上真正的業務規則
- 再把 API contract 文件化
- 再把正式入口收斂回主線
- 再升級資料庫
- 最後才升級正式 auth

### 結論四

這個專案最有教學價值的地方，不是它功能多，而是它保留了清楚的版本痕跡，能讓看到系統如何一步一步長大。

## 8. 一段可直接口頭講解的總結

這個專案的演進很適合拿來說明一個重要觀念：系統不是一開始就完整設計好，而是在一次次問題暴露後，透過迭代逐步長成。初版先跑通點餐流程；第二版處理資料不能保存的問題；第三版補足前端展示所需的資料欄位；第四版修正實際使用與部署時的體驗落差；第五版才真正進入使用者、訂單歸屬與歷史紀錄這種較接近真實產品的設計；第六版把既有 route schema 輸出成 OpenAPI / Swagger 文件；第七版再把這條主線收斂成目前實際執行的 `backend.ts`。這就是軟體迭代最值得理解的地方。
