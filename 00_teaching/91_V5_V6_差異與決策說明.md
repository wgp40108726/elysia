## 文件目的

本文件整理 `V5 使用者登入與個人訂單版` 與 `V6 OpenAPI／Swagger 文件版` 的差異，並說明這些差異背後的決策邏輯。

---

## 一句話總結

- V5 重點：把「登入 + 訂單歸屬 + 歷史查詢」等業務能力做完整。
- V6 重點：在不改動核心業務流程的前提下，將既有 route schema 轉成可瀏覽、可驗證的 OpenAPI 文件。

---

## 版本定位（來自講義主線）

在專案版本定位中，講義明確定義：

- V5：加入使用者、登入、訂單歸屬與歷史

同時講義也指出：從 V5 開始，`shared/contracts.ts`、`store/Store.ts`、`store/index.ts`、`store/json/JsonFileStore.ts` 已收斂成共用主線，因此 V5/V6 差異主要集中在 backend 入口層。

## 3) 文件邊界控制：V6 隱藏非教學主線路由

V6 刻意將技術性或非主線路由從文件隱藏，避免 Swagger UI 被雜訊干擾：

- `OPTIONS *`（CORS 預檢）設定 `detail.hide: true`
- `GET *`（SPA fallback）設定 `detail.hide: true`
- OpenAPI 自身路徑也透過 `exclude.paths` 排除文件循環干擾

## 這些差異是怎麼決定的？（決策脈絡）

決策核心不是「再加功能」，而是「把已存在的 schema 變成團隊可見契約」。

講義在 V6 章節的問題定義是：

- 到了 V5，雖已有大量 params/query/body/response schema
- 但協作者仍需直接翻 source，契約可見性不足
- 因此 V6 要解決的是「schema 能不能被看見、被瀏覽、被當成共同契約」

這也直接導出 V6 的三個設計決策：

1. 導入 OpenAPI plugin：把 schema 外顯成可瀏覽文件
2. 補齊 route metadata：讓文件有分類、語意、上下文
3. 隱藏非主線路由：提高教學與協作訊噪比

## 對教學與協作的實際影響

V5 到 V6 的演進，代表專案從「開發者內部可理解」走向「團隊外部可共享」：

- 教學上：不必先讀完整 backend 才能理解 API 面
- 協作上：前端、測試、文件可以對齊同一份 contract
- 維護上：API 變更更容易被看見，降低文件與實作脫節風險

---

## 最終結論

V6 相對 V5 的本質差異是：

- 不是新增一批新業務功能
- 而是把 V5 已成熟的 API schema，升級成可被瀏覽、分類、測試、共享的契約文件系統

所以 V6 的決策可視為「工程可運作」之後，朝「工程可溝通、可維護、可教學」推進的一次關鍵收斂。
