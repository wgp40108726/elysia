## 1. 00:00-03:00 破題：為什麼要談 Session

### 教師台詞（可照念）

- 「HTTP 是無狀態協定。也就是說，Server 不會自動記得你是誰。」
- 「你剛剛登入成功，不代表下一個請求 Server 就知道還是同一個人。」
- 「所以我們需要 session 機制，讓身份可以跨請求被可信地辨識。」

### 板書關鍵字

- `HTTP = stateless`
- `session = cross-request identity`
- `authentication` vs `authorization`

### 追問（點 1-2 位學生）

- 「只有登入成功一次，能不能保證之後每次請求都安全？」

---

## 2. 03:00-08:00 回到目前專案：它現在怎麼做

### 教師台詞（可照念）

- 「我們目前專案是教學簡化版流程。」
- 「登入後，後端回傳 `SessionUser`，前端放到 localStorage。」
- 「接下來查訂單時，前端會把 `userId` 帶到 API。」
- 「這個設計很適合教學起步，因為資料流很好懂、功能很快串起來。」

### 立即補充（非常重要）

- 「但這不是生產環境安全做法。」
- 「它的價值是教學，不是最終安全形態。」

### 板書流程

1. login -> 回 `SessionUser`
2. 存 localStorage
3. 帶 `userId` 呼叫 API
4. 後端查資料

## 3. 08:00-12:00 弱點拆解：為什麼這樣會有風險

### 教師台詞（可照念）

- 「第一個問題，前端資料可被改。localStorage 不是信任邊界。」
- 「第二個問題，如果後端只信 `userId`，就有機會越權。」
- 「第三個問題，沒有集中 session 治理，很難做強制登出與撤銷。」
- 「第四個問題，若發生 XSS，localStorage 內容可能被讀走。」

### 示範提問

- 「如果我把 URL 的 `userId=0001` 改成 `0002`，理論上應該發生什麼？」
- 「你希望系統回 200 還是 403？為什麼？」

### 教學訊息（要講）

- 「重點不是把前端寫得多漂亮，而是後端永遠不能把身份驗證責任交給前端輸入。」

### 補充 3 分鐘：CSRF 與 XSS 怎麼講

#### 教師台詞（可照念）

- 「CSRF 跟 XSS 常被混在一起，但本質不同。」
- 「CSRF 是借你的瀏覽器身份去送你不想送的請求。」
- 「XSS 是在你的頁面裡執行攻擊者的 JavaScript。」

#### 口語例子

- 「CSRF 像是有人偷拿你已登入的門禁卡去刷門。」
- 「XSS 像是有人直接進你辦公室，在你電腦前操作。」

#### 對照重點（板書）

#### 單頁比較圖使用提示（30 秒）

- 「請切到『單頁比較圖』：先看機制（Stateful/JWT），再看載體（Cookie/localStorage），最後看攻擊面（CSRF/XSS）與 Better Auth 的定位。」

2. XSS 防護：輸入驗證、輸出編碼、避免危險 DOM API、CSP
3. 連回憑證載體選擇：

- 「Cookie 要防 CSRF」
- 「localStorage 要非常小心 XSS」

## 4. 12:00-17:00 典型做法 A：Stateful Session

### 教師台詞（可照念）

- 「Stateful 的核心是：session 狀態放在伺服器。」
- 「前端通常只拿到 session id（常見在 Cookie）。」
- 「每次請求 Server 會用 session id 去查 session store，確認你是誰、是否仍有效。」

### 優點（講重點）

1. 可立即失效（封鎖、強制登出）
2. 權限更新可即時反映
3. 治理、稽核、管理通常比較直觀

### 缺點（講重點）

1. 需要 session store（如 Redis/DB）
2. 分散式部署要處理共享狀態
3. 維運負擔上升

### 一句話總結

- 「Stateful 偏向可治理性強，但要付出基礎設施成本。」

---

## 5. 17:00-21:00 典型做法 B：JWT Token

### 教師台詞（可照念）

- 「JWT 常見於 Stateless 思路：後端簽 token，前端帶 token。」
- 「後端驗簽即可知道使用者身份，不一定每次查 session store。」

### 優點（講重點）

1. 水平擴展友善
2. 跨服務傳遞身份資訊方便
3. API 化與微服務情境常見

### 缺點（講重點）

1. 撤銷與強制登出較麻煩
2. 需要嚴謹的到期與輪替策略
3. 若儲存/傳輸不當，風險很高

### 一句話總結

- 「JWT 偏向擴展性強，但安全治理要設計得更完整。」

---

## 6. 21:00-24:00 比較與決策：沒有神解，只有取捨

### 教師台詞（可照念）

- 「Session 與 JWT 不是誰絕對比較好，而是看你的風險模型與系統型態。」
- 「若你最在意可治理性與強制撤銷，Stateful 常更直觀。」
- 「若你最在意跨服務擴展與 API 生態，JWT 常更常見。」

### 快問快答

- 「單體後台系統你會先選哪個？為什麼？」
- 「多服務架構你會先選哪個？需要補什麼防護？」

## 7. 24:00-27:00 帶入 Better Auth：為什麼現在不手刻 JWT

### 教師台詞（可照念）

- 「我們接下來要導入 Better Auth，所以現在不急著手刻 JWT。」
- 「不是因為 JWT 不重要，而是要把精力放在更高價值的能力：授權規則、業務邏輯、整體架構。」
- 「Better Auth 幫我們把最容易做錯的 auth 基礎工程標準化。」

### 請講清楚它代替了什麼

1. session/token 的標準流程
2. 安全 cookie 的常見設定
3. OAuth provider（如 Google）整合
4. session/account/user 關聯與生命週期處理

### 要強調的教學訊息

- 「框架不是魔法，而是把業界反覆踩坑後的最佳實務產品化。」

### 補充 1 分鐘：Better Auth 責任邊界

#### 教師台詞（可照念）

- 「Better Auth 很強，但它不是萬靈丹。」
- 「它能幫你把身份機制標準化，例如 session/token 流程、cookie 安全預設、OAuth 整合。」
- 「但它不能替你寫完應用層安全，像是輸入驗證、輸出編碼、CSP、以及『A 不能看 B 資料』這種授權規則。」

#### 板書一句話

- 「Better Auth 負責 identity plumbing；團隊負責 app-level security。」

## 10. 教師備忘（避免常見誤教）

- 不要把「目前 demo 可用」講成「已符合上線安全」
- 不要把 JWT 神化成萬用解
- 不要只講 API 名稱，忽略風險邊界與治理策略
- 一定要反覆提醒：
  - 前端輸入不可直接信任
  - 身份驗證責任在後端
  - session/token 是機制，不是目的
