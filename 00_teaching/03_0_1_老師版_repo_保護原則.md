# 老師版 repo 保護原則

這份文件不是教學生怎麼 clone，而是教老師怎麼保護自己的教學主 repo，避免：

- 學生誤以為 public repo 可以直接 push
- 學生把老師 repo 當成自己的開發 repo
- 共用電腦上殘留老師的 GitHub credential
- `main` 被直接推入未審查內容

最核心的一句話是：

```text
public 只代表看得到、clone 得到，不代表寫得回去
```

---

## 1. 先釐清基本安全模型

學生就算可以：

- 看老師的 public repo
- clone 老師的 public repo
- fork 老師的 public repo

也不代表他可以直接 push 回老師的 repo。

要能 push 回老師 repo，至少要滿足其中一個條件：

1. 學生有老師 repo 的寫入權限
2. 學生拿到老師有效的 GitHub credential

所以真正的風險通常不是 `clone` 本身，而是：

- 老師把學生加成 collaborator
- 老師在學生可接觸的電腦上留下 PAT / SSH key / 已登入 session

---

## 2. 第一原則：不要給學生寫入權限

若老師的 repo 是教學主線 repo，預設就不應讓學生直接具有 write 權限。

### 最安全的預設做法

- 學生只 clone
- 學生只 fork，或建立自己的 repo
- 學生 push 到自己的 repo
- 老師若要收作業，看 PR 或看學生 repo

### 若是個人帳號 repo

個人帳號 repo 的權限模型很粗：

- owner
- collaborator

只要是 collaborator，基本上就有 push 權限。

所以：

- 不要把學生加成 collaborator
- 一旦加了，就等於開放寫入

---

## 3. 第二原則：一定保護 `main`

就算學生理論上不該有寫入權限，`main` 還是應該保護。

建議至少開啟：

1. 禁止直接 push 到 `main`
2. 只能透過 Pull Request 合併
3. 禁止 force push
4. 禁止刪除 branch

如果 GitHub 方案允許，也建議再加：

5. Require status checks
6. Require at least one approval
7. Restrict who can push to matching branches

這樣就算有人真的拿到 write 權限，也不容易直接污染主線。

---

## 4. 第三原則：不要在學生機器上留下老師 credential

這是教學現場最容易被忽略的風險。

若老師曾在：

- 教室公用電腦
- 學生自己的電腦
- 借用的筆電
- 共用的 VS Code / GitHub Desktop

登入自己的 GitHub 帳號，就可能留下：

- GitHub session
- credential manager 記憶的 token
- PAT
- SSH private key

一旦這些還有效，學生即使不是 collaborator，也可能借用老師身分 push。

### 老師應避免

- 不要在學生電腦上長期登入自己的 GitHub
- 不要把 PAT 直接貼在可回看的記事本或聊天室
- 不要把自己的 `~/.ssh` 整包帶到教室共用機器

### 若曾登入過，課後要確認

1. GitHub 已登出
2. Git credential helper 內沒有殘留老師憑證
3. 不再保留老師 PAT
4. SSH key 已移除或使用教學專用帳號

---

## 5. 第四原則：學生應該用自己的 repo，不應直接在老師 repo 上開發

老師應固定要求學生採用這個模型：

```text
upstream = 老師 repo
origin   = 學生自己的 repo
```

這樣的好處：

- 老師 repo 是教材基線
- 學生 repo 是個人作品與實驗空間
- 兩邊責任清楚

若學生沒有自己的 repo，就容易發生兩種問題：

1. 只會 clone，但不會建立自己的版本
2. 搞不清楚自己到底在推誰的 repo

所以學生版配套文件應一起給：

- `03_0_0_從老師_repo_建立自己的_repo_並持續同步上游.md`

---

## 6. 最適合課堂的大規模做法

如果學生很多，最穩的方案不是手動盯每個人 remote，而是直接用制度解決。

### 方案 A：最小可行做法

- 老師 repo 保持 public
- 不給學生 collaborator 權限
- 學生一律 clone 後建立自己的 repo
- 課堂固定檢查學生的 `git remote -v`

這適合：

- 小班
- 老師能手動檢查
- 不需要大量自動化收作業

### 方案 B：更正式的做法

- 使用 GitHub organization
- 老師 repo 放在 organization
- 學生只給 `Read`
- 需要交作業時用 PR 或獨立 repo

這樣比個人帳號 repo 更好管理，因為 organization 有較細的角色分級。

### 方案 C：班級很多時的最佳做法

- 使用 GitHub Classroom
- 老師 repo 當 template repo
- 每位學生接受 assignment 後，自動建立自己的 assignment repo

這樣幾乎可以從制度上避免：

- 學生共用同一個老師 repo
- 學生誤 push 到老師主線
- 老師手動幫每位學生建 repo

---

## 7. 課堂上應明講的幾句話

建議老師在一開始就講清楚：

```text
public 不等於可 push
老師 repo 是教材主線，不是學生直接開發的位置
學生自己的修改，應該推到自己的 repo
```

再加一句操作口訣：

```text
老師 repo 只拿來讀與同步
自己的 repo 才是拿來寫與 push
```

---

## 8. 若真的發生學生把內容推進老師 repo，先查什麼

若老師發現 repo 被污染，先不要急著判斷是 GitHub 壞掉。

先查三件事：

1. 這位學生是否被加成 collaborator
2. 這台機器是否殘留老師的 GitHub credential
3. `main` 是否根本沒有 branch protection

通常原因都在這三類之一。

---

## 9. 老師實際可執行的最小保護清單

若要用最少步驟保護 repo，建議至少做到：

1. 老師 repo 不加學生為 collaborator
2. `main` 開 branch protection
3. 課堂明講 `upstream = 老師`、`origin = 自己`
4. 要求學生第一天就建立自己的空 repo
5. 不在學生可碰到的機器上保留老師 credential

如果能再往上升級，下一階段是：

6. 改用 organization
7. 再進一步改用 GitHub Classroom
