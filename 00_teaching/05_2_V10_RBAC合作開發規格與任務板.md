## 一、摘要

本報告說明早餐店點餐系統在 V10 版本中導入 RBAC（Role-Based Access Control，角色權限控制）的設計方式。系統原本已具備菜單管理、訂單系統與 Better Auth + Google OAuth 登入功能，但不同使用者登入後仍需要更清楚的權限區分。

本次 V10 期末報告版本先專注在課程範圍內的角色權限管理，將使用者分為顧客、櫃台店員、廚師、店長/老闆與系統管理員五種角色。同時依照講義必做項目，納入 V10.1 基礎菜單版本化，讓菜單價格與內容變更可以被記錄與追蹤。

在時間有限的情況下，本報告選擇「版本歷史查詢 API」與「價格變動提示 UI」作為短時間內可完成、展示效果明確的延伸功能。客服系統、促銷系統與顯示編號重新排序則改列為後續版本規劃。

## 二、系統背景與問題

目前系統的基礎功能包含：

- Google OAuth 登入。
- 菜單瀏覽與管理。
- 訂單建立與狀態更新。
- PostgreSQL + Drizzle ORM 資料庫架構。
- Contracts、Route Schemas、Backend Routes 的分層設計。

在實際使用情境中，不同角色不應擁有相同操作權限。例如顧客只能查看自己的訂單，廚師只需要查看待製作訂單，店長可以管理菜單與審核內部人員申請，而系統管理員擁有最高權限。因此 V10 版本需要導入角色權限管理，以提高系統安全性與管理彈性。

## 三、V10 設計目標

1. 建立清楚的角色分類。
2. 讓後端 API 能依角色進行權限檢查。
3. 讓顧客只能操作與自己相關的資料。
4. 讓櫃台店員可以處理前台訂單。
5. 讓廚師可以查看與更新待製作訂單。
6. 讓店長/老闆可以管理菜單並審核內部人員申請。
7. 讓系統管理員保有完整管理權限。
8. 讓前端依照角色顯示不同操作入口。
9. 完成講義必做的基礎菜單版本化。
10. 提供菜單版本歷史查詢 API。
11. 在前端提供價格變動提示 UI。
12. 規劃超過 10 個 REST API，讓期末成果能清楚展示系統功能完整度。

## 六、設計原則

本專案採用「事實層級」的方式進行設計：

| 層級 | 說明 | 對應內容 |
| --- | --- | --- |
| 第1事實 | 業務領域中真實存在的概念 | 顧客、訂單、菜單、角色、菜單版本 |
| 第2事實 | 操作流程與 API 規格 | 建立訂單、更新訂單、角色申請、角色審核、查詢版本歷史 |
| 第3事實 | 實作細節 | Drizzle Schema、Guard Helper、Route Handler、Frontend Component |

此設計原則可以避免直接從程式碼開始修改，導致權限邏輯分散或前後端規格不一致。

## 九、帳號角色申請流程

系統預設登入使用者為 `customer`。如果使用者是櫃台人員、廚師或店長，可以透過「帳號申請更換為內部人員」提出申請。

流程如下：

1. 使用者登入後送出角色申請。
2. 系統建立一筆角色申請紀錄。
3. 店長/老闆或系統管理員查看申請。
4. 審核通過後，使用者取得對應角色。
5. 審核拒絕時，使用者仍維持原本角色。

系統管理員可以直接修改使用者角色，但店長/老闆不應直接指派 `admin` 權限。

由於系統採用多角色設計，內部人員也可以同時保留 `customer` 角色。例如廚師可以用顧客身分建立自己的訂單，但不能用廚師權限修改顧客訂單內容。櫃台店員可以代替顧客建立訂單，但訂單需要標註「由櫃台代建」，並記錄實際顧客與建立者。

## 十、V10 API 規劃

為了符合期末專案對 API 數量的期待，本次 V10 不額外塞入促銷或客服等大型功能，而是將 RBAC、角色申請、菜單版本化與訂單權限拆成清楚的 REST API。預計規劃 14 個 API，其中部分為既有 API 的權限化改造，部分為新增 API。

| 編號 | Method | API | 功能 | 權限 |
| --- | --- | --- | --- | --- |
| 1 | `GET` | `/api/me` | 取得目前登入使用者與角色 | 已登入 |
| 2 | `GET` | `/api/permissions/me` | 取得目前使用者可用權限清單 | 已登入 |
| 3 | `POST` | `/api/role-requests` | 申請更換為內部人員角色 | 已登入 |
| 4 | `GET` | `/api/role-requests` | 查看角色申請列表 | owner/admin |
| 5 | `GET` | `/api/role-requests/:id` | 查看單筆角色申請詳情 | owner/admin |
| 6 | `PATCH` | `/api/role-requests/:id` | 核准或拒絕角色申請 | owner/admin |
| 7 | `PATCH` | `/api/users/:id/roles` | 直接修改使用者角色 | admin |
| 8 | `GET` | `/api/menu` | 查看目前有效菜單與價格變動提示欄位 | 公開/已登入 |
| 9 | `POST` | `/api/menu` | 新增菜單項目 | owner/admin |
| 10 | `PATCH` | `/api/menu/:id` | 修改菜單並產生新版本 | owner/admin |
| 11 | `DELETE` | `/api/menu/:id` | 停用或刪除菜單項目 | owner/admin |
| 12 | `GET` | `/api/menu/:id/history` | 查看菜單版本歷史 | staff/owner/admin |
| 13 | `GET` | `/api/orders` | 依角色查看訂單列表 | customer/staff/chef/owner/admin |
| 14 | `PATCH` | `/api/orders/:id/status` | 更新訂單狀態 | staff/chef/owner/admin |

其中 API 8、9、10、11、13、14 可能已有對應功能，本次重點是補上角色權限與版本化行為；API 1 到 7、12 則是 RBAC 與版本歷史查詢所需的新增或明確化 API。

## 十四、後續版本規劃

### V11 個人延伸功能：客服系統

客服功能不納入 V10 期末報告主線，改列為 V11 個人延伸功能。這樣可以讓 V10 報告維持在課程要求的 RBAC 權限設計與基礎版本化，同時保留我自己想做的客服系統方向。

#### V11 設計動機

早餐店系統除了點餐與訂單管理，也可能需要處理顧客問題，例如訂單內容確認、餐點狀態詢問、餐點錯誤回報或其他售後問題。因此 V11 可以新增客服案件系統，讓顧客與店家有正式的溝通紀錄。

#### V11 角色規劃

V11 初期不新增獨立客服角色，而是先由 `staff` 櫃台店員兼任客服人員。原因是實際早餐店規模較小時，櫃台人員通常同時負責接單、回應顧客與處理現場問題。

#### V11 功能草案

| 功能 | 顧客 | 櫃台/客服 | 店長/老闆 | 系統管理員 |
| --- | --- | --- | --- | --- |
| 建立客服案件 | 允許 | 允許 | 允許 | 允許 |
| 查看客服案件 | 自己的案件 | 允許 | 允許 | 允許 |
| 回覆客服案件 | 不允許 | 允許 | 允許 | 允許 |
| 關閉客服案件 | 不允許 | 允許 | 允許 | 允許 |
| 查看客服統計 | 不允許 | 不允許 | 允許 | 允許 |

#### V11 實作方向

- 新增 `support_tickets` table。
- 定義 `supportTicketSchema`。
- 定義客服案件建立、回覆與狀態更新 API。
- 顧客只能查看自己的客服案件。
- 櫃台/客服、店長/老闆與系統管理員可以處理客服案件。
- 前端新增客服案件列表、案件詳情與回覆介面。

### V12 或後續：顯示編號重新排序

顯示編號重新排序可以透過 `menu_display_order` 表處理，讓同一個菜單實體在不同時間有不同顯示編號。例如某個商品原本是 1 號餐，後來因菜單整理改為 3 號餐。這個功能需要額外資料表、有效期間與管理 UI，時間成本較高，因此不放入本次期末實作。

### V13 或後續：促銷系統整合

促銷系統需要處理折扣規則、活動期間、適用商品、訂單價格計算與歷史追溯。這會影響訂單總額與付款語意，屬於較大型功能，因此不適合在期末時間緊湊時納入。

## 十五、結論

V10 版本的重點是讓早餐店系統從單純的登入功能，進一步升級為具有角色分工與權限控管的系統。透過 RBAC 設計，顧客、櫃台店員、廚師、店長/老闆與系統管理員可以各自取得符合業務需求的操作範圍。

在期末時間有限的情況下，本報告選擇完成講義必做的基礎 RBAC 與基礎菜單版本化，並加入版本歷史查詢 API 與價格變動提示 UI 作為可展示成果。客服系統、顯示編號重新排序與促銷系統則列為後續版本，避免 V10 範圍過大而影響完成度。
