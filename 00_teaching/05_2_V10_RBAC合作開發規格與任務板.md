# V10 RBAC 合作開發規格與任務板

來源講義：`00_teaching/05_1_V10_RBAC權限系統設計與實作講義.md`

工作分支：`feat/v10-rbac-collab`

## 目標

在 V9 Better Auth + Google OAuth 的基礎上，實作 V10 RBAC 權限系統。這份文件不是講義，而是我們共同開發時使用的工作板：先確認需求，再依序修改 contracts、database schema、auth mapping、guards、backend routes、frontend UI，最後補測試與部署筆記。

## 開發原則

- 第1事實：先定義業務角色、權限語意與資料模型。
- 第2事實：再定義 API 行為、輸入輸出與授權規則。
- 第3事實：最後才落到 Drizzle schema、migration、guard helper、route handler 與前端元件。
- 後端授權是安全邊界，前端權限控制只負責 UX。
- 先做 V10 MVP，避免一次把報表、庫存、促銷、進階管理全部塞進來。

## V10 MVP 範圍

### 必做

- 使用者具有一個或多個角色。
- 預設登入使用者為 `customer`。
- 後端能從 session 取得目前使用者與角色。
- API route 可以宣告並執行角色權限檢查。
- Owner/Admin 可以查看與處理角色申請。
- 顧客只能操作自己的訂單。
- Staff/Chef/Owner/Admin 依角色取得較高權限。
- 前端依角色顯示或隱藏操作入口。

### 暫不做

- 完整報表與營運分析。
- 庫存管理。
- 促銷與優惠系統。
- 複雜組織階層與多店分店管理。
- 細粒度 permission table。

## 角色草案

| Role | 中文 | 核心職責 |
| --- | --- | --- |
| `customer` | 顧客 | 查看菜單、建立訂單、查看自己的訂單 |
| `staff` | 櫃台店員 | 協助點餐、查看與處理前台訂單 |
| `chef` | 廚師 | 查看待製作訂單、更新製作狀態 |
| `owner` | 店長/老闆 | 管理菜單、查看所有訂單、核准角色申請 |
| `admin` | 系統管理員 | 系統級管理、角色管理、完整權限 |

## 權限矩陣草案

| 功能 | 顧客 | 櫃台店員 | 廚師 | 店長/老闆 | 系統管理員 |
| --- | --- | --- | --- | --- | --- |
| 查看菜單 | yes | yes | yes | yes | yes |
| 建立自己的訂單 | yes | yes | no | yes | yes |
| 查看自己的訂單 | yes | yes | no | yes | yes |
| 查看所有訂單 | no | yes | yes | yes | yes |
| 更新訂單狀態 | no | yes | yes | yes | yes |
| 修改顧客/自己的訂單 | yes | yes | no | yes | yes |
| 建立/修改菜單 | no | no | no | yes | yes |
| 刪除菜單 | no | no | no | yes | yes |
| 帳號申請更換為內部人員 | yes | yes | yes | yes | yes |
| 核准帳號更換為內部人員申請 | no | no | no | yes | yes |
| 直接修改使用者角色 | no | no | no | no | yes |

## 待確認問題

- [ ] 使用者是單角色還是多角色？講義傾向簡化為 array，本分支先採多角色 array，除非後續改成 many-to-many。
- [ ] `owner` 是否能指派 `admin`？建議不能，只有 `admin` 能給 `admin`。
- [ ] `staff` 是否能代替顧客建立訂單？若可以，訂單需要記錄 `createdByUserId` 與 `customerUserId`。
- [ ] `chef` 可否看到顧客資訊？建議只看製作需要的訂單摘要。
- [ ] 菜單版本化是否要和 RBAC 同時實作？建議 V10 先完成 RBAC，菜單版本化另開 V10.1。

## 實作任務板

### Phase 1: Contracts 層

- [ ] 在 `shared/contracts.ts` 定義 `userRoleSchema`。
- [ ] 定義 `currentUserSchema`，包含 `id`、`name`、`email`、`roles`。
- [ ] 定義 `roleRequestSchema`。
- [ ] 定義權限相關 enum 或 const map，避免 route 裡散落字串。

### Phase 2: Route Schemas 層

- [ ] 在 `shared/route-schemas.ts` 定義角色申請 body。
- [ ] 定義核准/拒絕角色申請 body。
- [ ] 定義角色管理 API response schema。
- [ ] 檢查既有訂單與菜單 route 是否需要補授權語意。

### Phase 3: Database 層

- [ ] 在 `db/schema.ts` 擴充 user roles 欄位。
- [ ] 新增 `role_requests` table。
- [ ] 產生 Drizzle migration。
- [ ] 準備第一位 admin 的 seed 或 bootstrap 策略。

### Phase 4: Auth Mapping 層

- [ ] 檢查 `auth/better-auth.ts` session 是否可取得 user id。
- [ ] 修改 `auth/user-mapper.ts`，把 DB user 轉成 app current user。
- [ ] 確保未設定角色的舊使用者會 fallback 為 `customer`。

### Phase 5: Guards 層

- [ ] 新增 `auth/guards.ts` 或依現有架構放置 guard helper。
- [ ] 實作 `requireUser`。
- [ ] 實作 `requireRole`。
- [ ] 實作 `requireAnyRole`。
- [ ] 實作訂單 owner 檢查，例如 `canAccessOrder`。

### Phase 6: Backend Routes

- [ ] 保護菜單寫入 API：owner/admin。
- [ ] 保護訂單列表 API：customer 只看自己的，staff/chef/owner/admin 可看授權範圍。
- [ ] 保護訂單狀態更新 API：staff/chef/owner/admin。
- [ ] 新增 `GET /api/me` 或確認既有 session endpoint 足夠前端使用。
- [ ] 新增角色申請 API。
- [ ] 新增角色申請審核 API。

### Phase 7: Frontend

- [ ] 擴充 auth state，讓 UI 可讀取 roles。
- [ ] 建立 `Can` 或 `RequireRole` 元件。
- [ ] 菜單管理按鈕只對 owner/admin 顯示。
- [ ] 訂單管理畫面依角色顯示不同視圖。
- [ ] 新增角色申請入口。
- [ ] 新增 owner/admin 角色審核入口。

### Phase 8: 測試與驗證

- [ ] customer 無法建立/修改菜單。
- [ ] customer 無法查看別人的訂單。
- [ ] staff 可以處理前台訂單但不能管理角色。
- [ ] chef 可以查看待製作訂單但不能管理菜單。
- [ ] owner 可以管理菜單與審核角色申請。
- [ ] admin 可以管理所有角色。
- [ ] 未登入使用者只能使用公開 API。

## 我們下一步可以怎麼開發

1. 先盤點目前 V9 的 auth、schema、route 寫法。
2. 選定單角色或多角色資料模型。
3. 從 contracts 開始補 RBAC 型別。
4. 接著做 DB schema 與 migration。
5. 最後把 guard 套進 backend route，前端再接角色 UI。
