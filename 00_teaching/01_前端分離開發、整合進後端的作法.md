# 前端分離開發、整合進後端的作法

## 1. 先說結論

本專案的前端整合進後端，不是靠額外搬檔，而是靠 `Vite build` 時直接把輸出目錄指定到後端會提供的靜態資源資料夾。

換句話說：

- 開發時，前後端分開跑
- build 時，前端產物直接輸出到根目錄 `public/`
- 部署時，由後端直接提供 `public/` 內的靜態檔案

---

## 2. 這個流程的三個關鍵點

### 2.1 前端 build 輸出目錄

在 `frontend/vite.config.ts` 中，`outDir` 會設成 `../public`。

這代表：

- `vite.config.ts` 位於 `frontend/`
- `../public` 會對應到專案根目錄的 `public/`
- 前端 build 完成後，產物會直接寫到後端可服務的靜態目錄

所以這一步不是「先 build 到 frontend/dist，再手動搬到後端」，而是「build 時就直接輸出到正確位置」。

### 2.2 觸發 build 的腳本

在根目錄 `package.json` 中，前端 build 會透過類似下面的腳本執行：

```json
{
  "scripts": {
    "build:frontend": "cd frontend && bun run build"
  }
}
```

這表示：

- 先切到 `frontend/`
- 再執行該目錄下的 Vite build
- 產物會依照 `vite.config.ts` 的 `outDir` 設定，直接輸出到根目錄 `public/`

### 2.3 後端靜態檔服務

在 `backend.ts` 中，後端會透過 `@elysiajs/static` 或等效設定，直接把 `public/` 當成靜態資源來源。

這表示：

- 前端 build 完成後，不需要再另外搬檔
- 後端啟動時，就能直接提供 build 出來的 HTML、JS、CSS 與圖片
- 部署時只要啟動後端服務，即可同時提供 API 與前端頁面

---

## 3. 補充說明

### 3.1 `emptyOutDir: true` 的作用

若 `vite.config.ts` 中設定了 `emptyOutDir: true`，表示每次 build 前都會先清空輸出目錄。

這樣做的好處是：

- 避免舊版前端資產殘留
- 避免 build 後混入過期檔案
- 減少部署時的靜態資源混亂

### 3.2 這種做法的教學價值

這個架構很適合教學，因為它同時保留了兩種能力：

- 開發時，可以清楚區分前端與後端責任
- 部署時，可以把系統收斂成單一後端入口

因此可以同時理解：

1. 什麼是前後端分離開發
2. 什麼是靜態資產整合部署
3. 為什麼 build 與 serving 是兩個不同階段

---

## 4. 一句話總結

本專案是用 `Vite` 的輸出目錄設定，把前端 build 產物直接放到後端的 `public/`，再由後端直接提供靜態檔案；核心不是搬檔，而是「在 build 階段就輸出到正確位置」。
